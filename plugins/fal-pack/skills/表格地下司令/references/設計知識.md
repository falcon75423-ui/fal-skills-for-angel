## 一、核心原則（直接搬 — Excel 場景仍成立）

### 1.1 多維 Hierarchy（不只字級）

單靠字級拉 hierarchy = 表格糊掉。Excel 場景至少兩個維度疊加：

| 維度 | 強層次 | 弱層次 |
|------|-------|-------|
| 字級 | 表頭 ≥ 資料 × 1.25 | 表頭 = 資料 |
| 字重 | Bold vs Regular | 同字重 |
| 底色 | 表頭明顯不同色 | 同色 |
| 邊框 | 表頭粗框 | 細框 / 無框 |
| 對齊 | 標題列置中 / 資料左右分離 | 全置中或全左對齊 |

**最佳實踐**：表頭列三維度同時動（字級 ↑ + Bold + 底色不同）。

### 1.2 字型 ≤ 3 套

一份 .xlsx 最多三套字型：
- 一套 Latin（**Arial** / Helvetica）
- 一套 CJK（**Noto Sans TC** / 思源黑體 / 微軟正黑體）
- 一套等寬數字（**Roboto Mono** / Consolas — 可選，數字密集表用）

> **反例**：Arial + Calibri + 微軟正黑體 + Times New Roman 四套混用 = 視覺風格分裂、印象散漫。
>
> **Google Sheets 安全鐵律**（landmines.md § 1）：英文用 Arial（不是 Calibri）、繁中用 Noto Sans TC。

### 1.3 Squint Test（瞇眼測試）

產出後做的視覺驗收工法。詳見 § 五自驗工法。

一句話：模糊處理後還能不能一眼看出 hierarchy。

### 1.4 絕對禁令清單（match-and-refuse rewrite）

對應 `landmines.md § 3 設計品味地雷` 的 8 條 ❌→✅。

精神：**看到就拒絕重寫**，不等客戶抱怨。本檔講設計動機，landmines.md 講「不准做什麼/該做什麼」。

### 1.5 DESIGN.md token + prose 雙介面思維

設計系統有兩種介面：
- **token**：機器讀的（Python `DESIGN_SYSTEM` dict / YAML）— 程式吃
- **prose**：人話描述的（markdown body）— 作者拍板用

**Excel 場景 v1 取捨**：
- `DESIGN_SYSTEM` dict（生成腳本開頭定義）已是 token 真相源
- v1 不做通用 `Excel_DESIGN.md.template`（YAGNI）— 作者通常不拍板字型/列高/邊框，只給內容由表哥設計
- v2 觸發條件（記在 SKILL.md 盲點段）：下次表哥碰到「使用者明確要求看設計規格文件」或「需要拍板字型/配色」的 Excel 案件時主動評估（不靠計數機制）
- 不滿足時不做

## 二、Excel 場景重寫（原則對但實作要換）

impeccable 原則用西文/Word/Web 邏輯設計，Excel 場景以下四條必改寫。

### 2.1 字級階層：Excel 字級節奏遠比 Word 平

**為什麼**：Excel 是「結構化格子網格」，字級 ratio ≥ 1.25 在 Word 場景成立，但 Excel 場景**列高 + 底色比字級更主導 hierarchy**。

| 角色 | 字級 (pt) | 跟下一層 ratio |
|------|----------|--------------|
| Title（封面標題）| 16 | × 1.333 |
| 區塊標題 | 12 | × 1.091 |
| 表頭 | 11 | × 1.100 |
| 內文 | 10 | × 1.111 |
| 註腳 | 9 | — |

**問題**：區塊標題 → 表頭 → 內文 ratio < 1.25。Word 視覺會糊，**Excel 用列高 + 底色 + 字重補足**。

**修正策略**：「字級 + 行高 + 底色」三維替代純字級 ratio。

### 2.2 列高 pt（不是行距倍數）

**Word**：行距 1.75 倍 = 字 11pt × 1.75 ≈ 19.25pt。
**Excel**：openpyxl `row_dimensions[r].height` 直接吃 pt 整數值。

| 角色 | 列高（pt）| 為什麼 |
|------|----------|-------|
| Title | 36 | 6×6，紙頁封面氣勢 |
| 區塊標題 | 30 | 6×5，比表頭高一階 |
| 表頭 | 24 | 6×4，跟內文做 hierarchy |
| 內文（中文 11pt 字）| 18 | 6×3，跟字級 1.64 倍對齊（接近 1.75 行距） |
| 內文（中文 10pt 字）| 18 | 6×3，密集表用 |
| 註腳 | 12 | 6×2，最緊 |
| 動態列高（合併儲存格 + wrap_text 長文）| 50-300+ | 任意 pt，**合法例外**（見 § 二.4）|

🔴 **絕對禁止**：列高用 7 / 11 / 13 / 17 pt 等非 6pt 整數倍 = 視覺節奏破碎。對應 `landmines.md § 3 設計品味地雷 #4` 的 ❌→✅ 對照（看到就拒絕重寫）。

### 2.3 行寬：Excel 單格 wrap_text 上限

**Word**：每行 28-35 中文字（A4 雙欄參考）。
**Excel**：合併儲存格 + wrap_text 場景下，max 35 CJK 字/列。

**為什麼**：CJK 字元等寬，可直接算字數。
- 28 字以下 = 太擠 / 視線回掃太頻繁
- 35 字以上 = 視線回掃距離過長 / 容易看錯行

**超過 35 字怎麼辦**：
- 拆分 cell（垂直方向多列）
- 或調寬欄寬（接 `auto_fit.py` 計算）

### 2.4 Vertical Rhythm：列高 6pt 整數倍

**為什麼用 6pt（不用 4pt）**：跟 F-017 文本參謀長 `typography_cjk.md` 對齊基礎單位（使用者 拍板跨 Skill 一致性 > 平台適配）。

選一個基礎單位，**全文垂直間距全是這個單位的整數倍**。

**Excel 6pt 系統**（典型值）：6 / 12 / 18 / 24 / 30 / 36 / 42 / 48 pt。

**例外（合法）**：
- **15pt**（Excel 預設列高 = 2.5×6 非整倍）— 接受預設值，`verify_design.py` 降為 WARNING 不 hard fail
- **動態列高 > 50pt + 合併儲存格**（wrap_text 長文場景）— 不限 pt，合法例外（B-3 揭露）

> **附**：4pt 系統（4 / 8 / 12 / 16 / 24）來自 impeccable spatial-design 借鑑——Excel 內建刻度其實多 4 倍數，但跨 Skill 一致性更重要。設計時若場景強烈需要 4pt 倍數，記入 SKILL.md 盲點段 v2 觸發點。

## 三、Excel 場景必砍（impeccable 有但 Excel 用不到）

| impeccable 原則 | 為什麼砍 |
|----------------|---------|
| ALL-CAPS 加字距（letter-spacing）| Excel 沒字元間距 API，要走 RichText 太貴 |

## 四、Excel 獨有設計判斷（impeccable 原檔沒寫）

### 4.1 欄寬節奏

**為什麼**：CJK 字元等寬可直接算字數，欄寬節奏靠「字元數倍數」建立視覺節奏。

| 內容類型 | 欄寬基礎倍數 | 典型值 |
|---------|------------|-------|
| CJK 內容欄 | 4 字元倍數 | 8 / 12 / 16 / 20 / 24 |
| 數字欄 | 6 字元倍數 | 6 / 12 / 18 |
| 日期欄 | 12 字元（yyyy/mm/dd 固定）| 12 |
| 短分類欄（A/B/C / Yes/No）| 6 字元 | 6 |

🔴 **絕對禁止**：欄寬用「2.3 字元 / 9.5 字元 / 14.7 字元」等異物——視覺自帶不安感。（無對應 landmines 條目——本條為設計知識獨有提醒，未來若觸發 ≥2 案次再升 landmines）

> **註**：以上「字元數」指**內容顯示寬度**（不含 padding）。實作時需加 2-3 字元 padding （見 `references/技術地雷.md` § CJK 字寬計算 — 欄寬計算公式）。

**計算工具**：`scripts/auto_fit.py`（CJK-aware）。

### 4.2 隔行底色（zebra striping）紀律

**為什麼**：奇偶列底色是「次序 hierarchy」載體，灰度差太大會搶「主視覺 hierarchy」（表頭/合計）。

| 灰度差 | 評級 | verify_design.py |
|-------|------|-----------------|
| ≤ 4% | 通過 | INFO |
| 4-8% | 警告 | WARNING |
| > 8% | 搶 hierarchy | BLOCKER |

**推薦配色**：
- `#FFFFFF` / `#F8F8F8`（差距 ≈ 3%）— 最佳
- `#FAFAF8` / `#F2F2F0`（差距 ≈ 3%）— 帶暖調

**反例**：`#FFFFFF` / `#E0E0E0`（差距 ≈ 12%）— 底色搶主視覺。

### 4.3 凍結窗格紀律

**為什麼**：表頭凍結列 = 視覺 hierarchy 的「重力錨點」，凍結位置即第一視覺停靠點。

**典型策略**：
| 表類型 | 凍結列 | 凍結欄 |
|-------|-------|-------|
| 一般資料表 | 第 1 列（表頭）| 第 1 欄（標題欄）|
| 多層表頭 | 資料起始列前一列 | 第 1-2 欄 |
| 跨期比較表 | 第 1 列 + 期別列 | 第 1 欄（指標名）|
| 純報告（無資料卷動）| 不凍結 | 不凍結 |

API：`ws.freeze_panes = 'B2'`（凍結 A 欄 + 1 列）。

### 4.4 數字對齊紀律

**為什麼**：同欄混用對齊 = 視覺安定感失敗（讀者潛意識察覺不一致但說不出哪裡不對）。

| 內容類型 | 對齊 | 數值格式碼 |
|---------|------|----------|
| 金額 | 右對齊 | `#,##0` |
| 百分比 | 右對齊 | `0.0%` |
| 比較欄（▲/▼/+/-）| 置中 | — |
| 日期 | 右對齊（書面）/ 左對齊（資料分析）| `yyyy/mm/dd` |
| 文字標籤 | 左對齊 | `@` |
| 短分類（Yes/No）| 置中 | — |

🔴 **絕對禁止**：同一欄內混用左右對齊（即使所有 cell 都是數字）。對應 `landmines.md § 3 設計品味地雷 #7`。

### 4.5 合併儲存格呼吸

**為什麼**：合併區邊界貼邊 = 視覺擠壓，內容像被擠進框裡。

| 設計要求 | 實作 |
|---------|-----|
| 四邊界 padding ≥ 2pt | cell.alignment indent / vertical=center |
| 四邊框完整 | `border_merged_range`（見 references/技術地雷.md） |
| 動態列高 50-300+ pt | wrap_text + 列高自動計算 |
| 合併不過度 | 一個合併區只表達一個概念，避免 5×5 巨大合併 |

### 4.6 條件格式 hierarchy

**為什麼**：條件格式是視覺最強的 hierarchy 工具——用錯會破壞整張表的層次感。

| 條件格式類型 | 推薦 | 禁忌 |
|------------|------|-----|
| Color Scale（色階）| 同色系明度差（如深藍 → 淺藍）| 跨色系混用（紅 → 黃 → 綠）|
| Data Bar（資料條）| 單色（如 `#0F7AA8`）| 漸層（Google Sheets 不支援）|
| Icon Set | 三色箭頭（▲/→/▼）| 紅綠點 IconSet（GS 渲染壞 + 色弱看不出）|
| 上下界突顯 | 字色變化 | 整 cell 底色變化（搶主視覺）|

🔴 **絕對禁止**：紅綠對比給色弱讀者（landmines.md § 3 #1）。

### 4.7 數值格式碼一致性

**為什麼**：同一欄不同格式碼 = 同樣是數字但顯示樣貌不同 = 讀者潛意識懷疑資料正確性。

| 場景 | 格式碼 | 一致性紀律 |
|------|-------|----------|
| 金額欄 | `#,##0` 或 `#,##0.00` | 全欄統一，不可一格 0 一格 0.00 |
| 百分比欄 | `0.0%` | 小數位數全欄統一 |
| 日期欄 | `yyyy/mm/dd` | 跨欄也統一 |
| 年份 | `@`（文字格式）或字串 | 必須——否則 2026 會顯示 2,026 |
| 零值顯示 | `#,##0;-#,##0;"-"` | 零顯示為「-」更清爽 |

### 4.8 列印佈局禁令

**為什麼**：Google Sheets 完全忽略 Excel 列印設定（margin / orientation / scale / page break / print_area）。

**規則**：
- 純 Excel 場景：列印標題列必設（`print_title_rows`）
- 跨平台場景（會匯到 GS）：**列印設定不可依賴**——必須在交付時告知使用者「列印請用 Excel 開啟，GS 列印會跑版」

## 五、自驗工法（dogfood）

每次產出 .xlsx 後跑驗證管線 Layer 1.5：

```bash
python scripts/verify_design.py path/to/file.xlsx
python scripts/verify_design.py path/to/file.xlsx --strict   # WARNING 也視為 fail
python scripts/verify_design.py path/to/file.xlsx --json     # 機器讀
```

**6 個 check 對應本檔章節**：

| Check | 對應章節 | landmines.md § 3 |
|-------|---------|-----------------|
| FONT_COUNT | § 一.2 字型 ≤ 3 | #3 |
| ZEBRA_CONTRAST | § 四.2 隔行底色 | #6 |
| ROW_HEIGHT_RHYTHM | § 二.4 列高 6pt 整倍數 | #4 |
| NUMBER_ALIGNMENT | § 四.4 數字對齊 | #7 |
| RED_GREEN_CONTRAST | § 四.6 條件格式 hierarchy | #1 |
| PURE_BLACK_WHITE | landmines #8 | #8 |

**退出碼**：0 = 全綠 / 1 = 有 WARNING / 2 = 有 BLOCKER。

### 5.1 Squint Test（瞇眼測試）— 視覺驗收

**為什麼要做**：讀者第一眼看到表格時，眼睛是失焦的——不是逐 cell 讀，是先掃整體 hierarchy。模糊看不到 hierarchy = 讀者迷路。

**操作**（接 `scripts/render_preview.py`）：

1. xlsx → Excel COM → PDF：`excel.ActiveWorkbook.ExportAsFixedFormat(0, pdf_path)`
2. PDF → PNG：`pdftoppm input.pdf output -png -r 72`
3. PNG 高斯模糊 5px（PIL `ImageFilter.GaussianBlur(5)`）
4. 表哥讀模糊版自答三題：
   - 最重要的元素能一眼看出嗎？
   - 第二重要的呢？
   - 區塊分組清楚嗎？

**通過標準**：三題全 Yes。任一模糊看不出 = hierarchy 失敗 → 回鍛打。

**例外**（呼應 SKILL.md 盲點段「過度工程化」）：
- ≤ 5 行的小表格跳過 Squint Test（模糊後剩一坨色塊看不出來）
- 純資料表（無 hierarchy 設計）跳過
