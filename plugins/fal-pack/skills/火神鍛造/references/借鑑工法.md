# 借鑑工法

> **讀取時機**：
> - 鍛造階段碰到「借鑑第三方智慧資本」時（兵器深審通過 + 決定 fork 進來時）
> - 觸發詞：fork / 借鑑 / 抄寫 / 翻譯第三方原則 / 中文化重寫第三方 reference
>
> **本檔性質**：把外部智慧資本搬進來的方法論——不是單純複製，是有選擇的吸收。
>
> **設計來源**：F-017 借 impeccable v3.0.5 時首次系統化（先前 F-003 / F-015 借 skillful-mcp / playwright-cli 各次零碎，沒結晶）。淬鍊師審視 F-017 後（2026-04-30）建議獨立成檔。

---

## 一、四象限分區（核心工法）

把第三方原則 / 知識 / 工具拆成四類，分類處理：

| 象限 | 動作 | 判別準則 |
|------|------|---------|
| **直接搬** | 不改照搬，引用來源 | 跨文化／跨技術棧仍成立的核心原則 |
| **改寫** | 保留靈魂改外殼 | 原則對但語法／單位／場景不對 |
| **砍掉** | 不搬不引用 | 在我們場景沒實質效益（甚至會誤導） |
| **新增** | 我們有它沒寫的 | 我們場景獨有的議題（補上對映性） |

---

## 二、各象限判別準則

### 2.1 直接搬

- **判別**：跨文化、跨技術棧、跨場景仍成立的設計原則
- **辨識訊號**：原則是「為什麼這樣做」（principle）而不是「怎麼做」（implementation）
- **F-017 範例**（借 impeccable typography）：Modular Scale ≥1.25 / 多維 hierarchy / 字型 ≤3 / 段落呼吸擇一 / Squint Test / ALL-CAPS 加字距

### 2.2 改寫

- **判別**：原則正確但實作細節要換場景
- **辨識訊號**：原檔給的是西文/web/JS 場景的具體值，套到中文/Word/Python 不對
- **F-017 範例**：
  - line-height: 1.5 → 中文行距 1.75-2.0
  - max-width: 65ch → 中文每行 28-35 字
  - vertical rhythm 24px → Word 段距 6pt 整數倍

### 2.3 砍掉

- **判別**：純技術棧細節 / 跨場景無實質效益 / 我們場景根本不存在的問題
- **辨識訊號**：搬過來會誤導（讓讀者以為這是我們也要做的事）
- **F-017 範例**：`font-display: swap` / `clamp()` / Container queries / OpenType features / variable fonts / `text-wrap` / ARIA 觸控 target

### 2.4 新增

- **判別**：我們的場景獨有但原檔沒寫的議題
- **辨識訊號**：「這在原檔找不到對應」+「我們場景必經」
- **F-017 範例**：CJK 雙字型 / 避頭尾 / 中英間距（pangu） / 標點壓縮 / 首行縮排傳統 vs 商業習慣

## 三、流程整合

### 觸發條件

鍛造案件需要 fork 第三方資產時，依序：

1. **兵器深審完成**（`references/兵器深審流程.md` / L0 安全掃描通過 / 兵器巡檢記錄登記）
2. **決定借進來**（不只觀察學習，要實際 fork 進 Skill）
3. **進入鍛打階段前先做四象限分區**

### 跑法

1. 列出第三方原檔的所有原則／工具／知識點（用條列形式逐項）
2. 對每點做四象限分類（多人鍛造時可分頭做後合併，每點要拍板理由）
3. 寫進借鑑檔的「來源致謝」段，標明四象限比例（如：直接搬 6 條 / 改寫 3 條 / 砍掉 7 條 / 新增 5 條）
4. 鍛打階段按分類處理：
   - **直接搬**：引用原文 + 翻譯（標清楚原檔出處）
   - **改寫**：保留邏輯結構，換場景細節（必須說明 why——為什麼我們場景這個值不同）
   - **砍掉**：不出現在借鑑檔（必要時可放「砍掉的原則」對照表給未來查，避免重新發明）
   - **新增**：自己寫，標明「原檔沒寫」

---

## 四、寫作格式建議

借鑑檔的章節結構，二選一：

### 結構 A（按四象限分章節）

- § 一 直接搬（核心原則跨場景仍適用）
- § 二 改寫（原則對但語法/單位/場景不對）
- § 三 砍掉（在我們場景沒實質效益）
- § 四 新增（我們場景獨有，原檔沒寫）

### 結構 B（合併描述）

- § 一 核心原則（直接適用） — 對應「直接搬」
- § 二 場景重寫 — 對應「改寫」
- § 三 必砍（場景無效益） — 對應「砍掉」
- § 四 場景獨有 — 對應「新增」

**F-017 typography_cjk.md 用結構 B**。經驗上結構 B 對讀者更友善（讀「核心原則」比讀「直接搬」直覺），但結構 A 對撰寫者更省力。

## 五、紀律

- 🔴 **直接搬必須引用來源**（致謝段標明出處：原檔 / commit hash / version）
- 🔴 **改寫必須說明 why**（不能只給新值，要說「為什麼中文場景需要 1.75 行距而不是西文 1.5」）
- 🔴 **砍掉的東西不要徹底消失**——記錄在「砍掉的原則」清單供未來查（避免重新發明已被砍掉的東西）
- 🔴 **新增的東西要在借鑑檔中標明「原檔沒寫」**（給未來讀者知道誰原創）
- 🔴 **借鑑檔自身要遵守單一真相源**——如果新增的內容已存在 Skill 內既有檔，引用既有檔不複製（F-017 § 7.2 黑線分工聲明 = 這條紀律的具體應用）
