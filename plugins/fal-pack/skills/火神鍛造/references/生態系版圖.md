選料時必問：這個新 Skill 在版圖裡的位置是什麼？跟誰協作？跟誰有邊界？

**已建立的跨 Skill 協作協議**：

| 協議 | 參與者 | 觸發 | 說明 |
|------|--------|------|------|
| **聯合研判** | 偵察兵 × 顧問團 | 「聯合研判 [主題]」 | 遞迴協作：偵察兵搜情報 → 顧問團分析找缺口 → 偵察兵補搜 → 收斂 |
| **表格品質閘門** | 賽事分析 CEO × 表哥 | CEO 產出 xlsx 後 / 覆盤更新後 / 模板需變更時 | 三場景握手：H1 日常驗證、H2 格式修復、H3 模板演化。CEO 管內容，表哥管格式工程 |
| **深度獵取委託** | 法眼 → 偵察兵 | 法眼說「這個方向我要深挖」 | 法眼傳委託格式（目標/深度/特別關注），偵察兵回傳來源清單+摘要+品質初判 |
| **Google 上傳交接** | 表哥 / 簡報大師 / 表弟 → Google工具箱 | 本地檔案產出後需上傳雲端 | 產出方完成檔案 → 引導使用者呼叫 Google工具箱上傳 |
| **出版商協議** | 任何內容方 → 表弟 | 「請出版商來處理」「交給出版商」 | 內容方傳 Brief → 表弟讀稿 → 排版對話 → 共同決議 → 產出 .docx → 視覺校樣 → 確認交付 |

**隱式呼叫機制**：Claude Code 中的 Skill 組合是隱式的——在 _index.md / SKILL.md 中 by name 引用其他 Skill，模型會自動呼叫（如果已安裝）。不需要顯式 import 或依賴宣告。

### 通用 Brief 格式（最低共同標準）

Skill 之間委託時，Brief 至少包含三要素：

```
【做什麼】一句話說清楚要委託的事
【為什麼】這個產出拿來做什麼（讓被委託方能判斷優先級和品質標準）
【回傳什麼】期望的交付物格式
```

各 Skill 可以在此基礎上擴展自己的 Brief 欄位（如表哥加 `xlsx 路徑`、偵察兵加 `深度等級`），但這三個不能少。沒有【為什麼】的 Brief = 被委託方無法判斷品質標準，容易做偏。

### 多 Skill 協調規則

當一個使用者請求需要多個 Skill 協作時（如「做一份有圖有表的簡報」）：

**核心原則：離使用者最近的 Skill 主導，其他是被呼叫的工具。**

判斷「誰離使用者最近」的方法：使用者的意圖最終要交付什麼？
- 交付物是簡報 → 簡報大師主導，視覺導演和表哥是工具
- 交付物是 Excel → 表哥主導
- 交付物是情報報告 → 偵察兵主導
- 交付物是決策 → 顧問團主導

**主導方的責任**：
1. 拆解任務，決定哪些環節需要委託
2. 用通用 Brief 格式向其他 Skill 發 Brief
3. 接收交付物，整合進最終產出
4. 對最終品質負責

**常見多 Skill 場景**：

| 場景 | 主導 | 被呼叫 | 協作方式 |
|------|------|--------|---------|
| 有圖有表的簡報 | 簡報大師 | 視覺導演（配圖）+ 表哥（數據表格） | 簡報大師設計結構 → 分別委託 → 整合 |
| 深度研究後做決策 | 顧問團 | 偵察兵（情報） | 聯合研判（已有專屬協議） |
| 情報精選存檔出刊 | 法眼 | 偵察兵（深度獵取）+ NotebookLM（存檔） | 法眼主導出刊節奏 |
| Excel 完成後上傳 | 表哥 | Google工具箱（上傳） | 表哥交付後交接 |

## 類型參考表

選料時用來快速定位新 Skill 屬於哪種建築類型：

| 類型 | 做什麼 | 典型特徵 |
|------|--------|---------|
| 工具/API 參考 | 教 Claude 正確使用某個工具或函式庫 | gotchas 多、程式碼片段、邊界案例 |
| 產品驗證 | 測試或驗證產出是否正確 | 搭配 Playwright/tmux 等外部工具、斷言腳本 |
| 資料撈取與分析 | 連接資料源、執行查詢、產出洞察 | 資料庫 schema、dashboard ID、查詢模式 |
| 流程自動化 | 把重複性工作流變成一個指令 | 依賴其他 Skill/MCP、log 檔追蹤歷史 |
| 腳手架/模板 | 生成特定框架的樣板程式碼 | assets 資料夾、boilerplate、config 預設 |
| 程式碼品質 | 強制或檢查程式碼標準 | 可搭配 hooks 自動執行、linter 整合 |
| 部署/CI | 建置、測試、部署、監控 PR | 多步驟流程、錯誤回滾邏輯 |
| 除錯手冊 | 從症狀出發，走調查流程，產出報告 | 症狀→工具→查詢模式 的對照表 |
| 基礎建設 | 維護性操作（清理、依賴管理、成本調查） | 涉及破壞性操作，需防護網 |

不是每個 Skill 都能乾淨歸類，但有了類型意識，選料時不會把展覽館的需求硬套進住宅的框架。
