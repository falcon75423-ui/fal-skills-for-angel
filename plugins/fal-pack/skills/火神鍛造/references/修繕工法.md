# 修繕工法

> 讀取時機：進入「翻修」或「修繕」路徑時載入。

---

## 翻修：系統性除錯五步

不猜，不補丁，追根因。

```
1. 症狀收集 — 什麼壞了？怎麼壞的？能穩定重現嗎？不能重現就不要猜
2. 根因調查 — 讀 landmines.md（踩過同樣的雷嗎？）
               → 問題在哪個層級？WHO / HOW / WHAT？
               → 最近改了什麼？（git diff 思維）
3. 模式比對 — 有沒有類似的 Skill 運作正常？差在哪？
4. 假說驗證 — 提出具體假說，寫下來。最小修改驗證。一次改一個變數
5. 三振出局 — 修三次沒好 → 不是 bug，是結構問題 → 回到選料重新問 WHO
```

> **鐵律**：症狀修復是失敗。沒找到根因就不動手。「看到問題，讓我修修看」→ 停。看到症狀 ≠ 理解根因。

---

## 解法層級判斷

同一個問題用不同層級修，壽命天差地遠。修之前先問：**我在用哪個層級？**

```
補丁層（一次性修復）
  「R40 有垃圾，清掉」
  → 壽命：一天。下次同樣的問題會在別處出現
  → 只在趕工或影響範圍極小時用。用了就標記：⚠️ 這是補丁，根因未解

規則層（加檢查步驟 / 加人工閘門）
  「術語速查：交付前掃描新術語」
  → 壽命：看記憶力。有人記得就有效，忘了就失效
  → 適合低頻、難以程式化的品質把關

結構層（改架構 / 程式化閘門 / 單一真相來源）
  「格式只在 design_system.py 定義一次」
  「驗證閘門寫進腳本，每次自動跑」
  → 壽命：直到架構再次改變。不需要記得，不需要判斷
  → 首選。成本高但一勞永逸
```

🔴 **預設選結構層**。只有明確理由（「太罕見不值得自動化」「一次性問題」）才降級到規則層或補丁層。

## 修繕協議（context 衛生）

在其他 Skill 的對話中發現問題時，不在現場改結構。

**流程**：
```
現場發現問題 → 「修繕」觸發深度討論（在當前對話，因為需要上下文）
→ 討論完 → 「製作修繕單」→ 儲存修繕單
→ 開新對話召喚火神 → 帶著修繕單進場
→ 工頭產出「施工評估」
→ 確認後施工 → 教訓寫入 landmines.md（免疫）
```

**修繕單 vs 施工評估 vs landmines.md**：
- **修繕單** = 輸入（問題描述、根因分析、討論結論、建議改法）— 診斷書，不是施工圖
- **施工評估** = 工頭的專業判斷 — 修繕單的建議改法在現有結構裡裝得進去嗎？有副作用嗎？有更根本的解法嗎？
- **landmines.md** = 輸出（修完後萃取的教訓）— 給未來 fal 看的免疫紀錄

**施工評估四種結論**：
- **採納**：改法可行，直接施工
- **深化**：改法方向對但不夠深，提出結構層替代方案
- **打回**：改法有副作用或盲點，帶理由回去跟提出方討論
- **拆分**：一張修繕單其實是多個獨立問題，拆開處理

**修繕單格式**：`[Skill目錄]/修繕單/YYYY-MM-DD_[問題摘要].md`，修完可歸檔或刪除。

**例外**：純粹的參數修正、錯字——現場改就好。但凡涉及「為什麼這裡會出錯」的結構思考，都回工廠。

## 🔴 施工差異記錄（現場調整的即時回寫）

> 修繕單是靜態文件，施工過程是動態的。現場調整沒有回寫 = 修繕單變成過期施工圖 = 驗收時對不上。

**觸發條件**：工頭在施工中做出任何偏離修繕單的調整——新增欄位、改計算邏輯、調整施工順序、變更架構決策——**改的當下就寫**，不是事後補。

**檔案位置**：修繕單同目錄，命名 `{修繕單名稱}_施工差異.md`。

**格式**：

```markdown
# 施工差異記錄

## {日期} {調整摘要}
- **原方案**：修繕單寫的是什麼
- **改為**：實際做了什麼
- **原因**：一句話
- **影響檔案**：哪些檔案被改了
```

**為什麼是必經齒輪**：
- 修繕單提交者（通常是執行股東）做驗收時，看的是「修繕單 + 差異記錄」
- 沒有差異記錄 = 驗收方不知道現場改了什麼 = 驗收形式化
- 差異記錄也是下次鍛造的教訓來源——「為什麼現場要改」本身就是設計規格的反饋

## 同類問題計數器

landmines.md 記錄已解決的地雷，但缺少**未升級的重複症狀**追蹤。Skill 有工具操作的，建議在 landmines.md 中加入活躍症狀追蹤區：

```
## 活躍症狀追蹤

| 症狀 | 首次出現 | 累計次數 | 上次出現 | 狀態 |
|------|---------|---------|---------|------|
| （範例）更新後格式斷裂 | 03-15 | 3 | 03-27 | 🔴 三振→待結構修繕 |
```

**閾值規則**：累計 ≥ 3 次 → 自動標記為結構問題 → 觸發修繕討論。每次遇到問題先查表，不是「想到才回頭看」。

## 🔴 一進一出原則（防止退化迴圈）

> 補丁 → 規則堆積 → context 膨脹 → 注意力 U 型流失 → 品質下降 → 更多補丁。這是退化迴圈。

每次修繕要加規則時，必須回答：

- **它取代了哪條舊規則？** → 替換，不是新增。總行數不變或減少
- **它暴露了一個需要結構層解法的問題？** → 升級為結構修繕，不是加規則
- 如果答案是「都不是，就是要多加一條」→ 🔴 紅旗。停下來想：是不是 WHO 層定義不夠，才需要這麼多 HOW 層規則來補？

## Context 體重秤（修繕時必做）

```
Context 體檢
├── _index.md 目前幾行？超過預算了嗎？（建議 ≤ 300 行）
├── 有沒有規則是「同一原則的不同說法」？→ 合併
├── 有沒有規則是「操作細節」而非「判斷原則」？→ 搬到 L3
├── 有沒有規則是「補丁的補丁」？→ 三振升級為結構修繕
└── 刪掉任一段，Skill 會斷嗎？→ 不會就刪
```

> 注意力 U 型曲線：context 開頭和結尾注意力最強，中間最弱。_index.md 越長，中間的操作細節和判斷規則——最需要被精確執行的東西——住在注意力最弱的地方。這是結構性的，不是靠「寫清楚一點」能解決的。只有瘦身和分層能解。

## 客戶服務協議

> 工頭是開店的商家，不是接案的個人。以下是經營鋪子的基本規矩。

### 🆕 修繕的兩種情境（依客戶身份；F-004 上線 2026-04-26）

「修繕」是通用動詞（修補/優化動作），依客戶身份**並列為兩種情境**，各有各的標準流程：

| 情境 | 客戶身份 | 結構特徵 | 標準流程 | 編號慣例 |
|------|---------|---------|---------|---------|
| **A 對外部客戶修繕** | 賽事分析 / 占星師 / 未來新客戶 | 客戶端有獨立開單者；需跨團隊驗收 | 接單流程 + 即時驗收雙簽協議（5 步）| 客戶自有編號（如 賽事分析 的 客戶 #XXX）|
| **B 對火神自身修繕/優化** | 火神鍛造（自己）| 工頭發現自身問題；使用者 是唯一真客戶 | 火神三件套（3 步：討論→拍板施工→Bridge+工作簿+看板）| F-XXX |

**核心原則**：兩種情境**並列不主從**。情境 A 的「修繕單檔案 + subagent 驗收 + 末尾雙簽」對情境 B 不適用——情境 B 不需要那些（使用者 拍板即親簽）。

### 接單流程（情境 A 適用）

> 情境 B 規格見本段末「火神自身修繕/優化規格」。

```
客戶端開修繕單 → 工頭讀單
→ 在客戶工作簿登記「🏗️ 施工中」（沒登記 = 沒接）
→ 順手更新 clients/_index.md 看板
→ 產出施工評估 → 施工 → 淬火
→ 🆕 召喚對方負責人驗收（subagent，無條件；詳見「即時驗收雙簽協議」）
→ 第一輪 PASS → 雙簽 / FAIL → 強制結構層根因分析 → 一次解決 → 第二輪必 PASS
→ 🆕 修繕單末尾寫「## 施工 + 驗收紀錄」段（含雙方證據 + 雙簽日期）
→ 🆕 單一完工事件三系統同步：修繕單 mv archived/ + 工作簿登記「✅ 完工」 + 看板更新
→ Bridge Document 存入匠神大道/鍛造記錄/
```

**關鍵變動**（F-004 上線 2026-04-26）：
- 完工流程從「異步驗收」（工頭完工 → 等開單者下對話來驗收）改為「**即時雙簽**」（工頭完工 → 當下召喚開單者 subagent → 雙簽 → 歸檔）
- 取消獨立「驗收紀錄.md」雙寫摩擦——驗收真相源就在修繕單末尾雙簽段
- 完工從「N 個獨立動作」（容易漏）→「**單一完工事件**」（雙簽當下三系統同步更新）

詳見下方「即時驗收雙簽協議」section。

### 工作簿存摺制

工作簿就像銀行存摺——一本有頁數上限，寫滿換新的。

- 每本上限 **20 筆**
- 寫滿時封存為 `工作簿_001.md`，開新的 `工作簿.md`
- 新本開頭寫「承上本結餘：X 案完工、Y 案進行中」
- 平常只讀最新那本，舊本需要時再打開

### 寫時更新看板

**不另外維護看板。** 每次更新任何客戶工作簿時，順手改 `clients/_index.md` 對應那一行。就像餐廳服務生送完菜順手在點菜單打勾。

### 斷點切分

施工中發現問題超出客戶範圍（例如發現工頭自身 bug）時：
1. 把當前客戶的事推進到一個自然切分點
2. 做成兩張單：客戶的繼續走、自己的另開
3. 客戶工作簿記一筆「施工中發現自身問題，已拆單處理」
4. 各自在不同對話處理

### 鍛造藍圖先行

從零鍛造新 Skill 時，選料之前先畫藍圖：
- 這個 Skill 會產生哪些檔案？
- 這些檔案要住在哪個目錄？
- 目錄結構 = 記憶的樹狀圖
- 藍圖確認後才建客戶檔案（直接完整版，不分兩階段）

### 工具選型

鍛造時的工具盤點要加一層判斷：同一個功能如果有多種工具可選（例如瀏覽器操作有 Chrome MCP、Playwright MCP、Claude in Chrome），必須評估哪個最適合這個 Skill，並在客戶工作簿記錄選擇理由。

### 🆕 即時驗收雙簽協議（情境 A 適用 / F-004 上線 2026-04-26）

> **適用範圍**：情境 A 對外部客戶修繕（賽事分析 / 占星師 / 未來新客戶）。
> 情境 B 火神自身修繕/優化規格見本段末「火神自身修繕/優化規格（情境 B）」。
>
> **取代**：原「異步驗收」模式（工頭完工 → 等開單者下對話來驗收）。
> **根因**：22 張完工未歸檔的歸檔欠債實證——「異步」+「多系統雙寫」+「讓工頭自評觸發條件」三重結構性失效。
> **解法**：對方負責人 subagent 即時驗收 + 雙簽 + 即時歸檔 = 單一完工事件三系統同步。

#### 對方負責人進場方式（依情境結構，非工頭主觀判斷）

| 情境本質 | 判別準則 | 進場方式 |
|---------|---------|---------|
| **執行接力** | 推進工作的下一步（沒有對立結構，同人接力）| **同對話切換**（角色宣告即可）|
| **驗收檢查** | 「我做完，請對方檢查」（對立結構：被驗證者 vs 驗證者）| 🔴 **subagent 無條件** |
| **跨專業諮詢** | 工頭沒這個專業眼光 | **subagent**（專業 Skill：表哥 / 表弟 / 瞇瞇眼 / 法眼 / 顧問團 / 簡報大師）|

**核心原則**：判斷依據是「**情境的結構**」（有對立關係嗎？需要外部專業嗎？），不是工頭主觀判斷「我污染了沒」。

**為什麼驗收 = subagent 無條件**：
- 工頭剛完成複雜修繕、寫了多處施工差異、自我論證已建構成形
- 同對話切換驗收 = 同人切換帽子 → 工頭視角的「這個方案合理」很難被自己推翻
- 「讓被污染者判斷自己污染程度」= meta-level 自指悖論
- 對齊 CLAUDE.md §八「閘門 = 結構性檢查」+ M02 退役教訓「規則層提醒靠記憶力 = 失效」

#### 驗收流程

```
工頭施工 + 淬火完成
  ↓
工頭呈交施工證據（grep + mtime + 結構驗證 + 反向隔離 + 修改清單）
  ↓
召喚對方負責人 subagent 進場（根據情境）：
  ├─ 賽事分析 修繕單 → 召喚開單者（賽事分析-fal / 季後賽 CEO / 例行賽 CEO）
  ├─ 火神鍛造自身單 → 直接呈 使用者 親簽（不是 subagent，是真客戶拍板）
  └─ 跨專業諮詢需求 → 召喚對應專業 Skill
  ↓
第一輪 subagent 驗收
  ├─ PASS → 雙簽歸檔，完工
  └─ FAIL → 🔴 強制觸發「結構層根因分析」（不允許「再修一輪小補丁」退化迴圈）
              ↓
            雙方負責人 + 可選 使用者 對齊根因問題清單：
            - 第一輪 FAIL 點是工頭施工不透徹？
            - 還是開單者開單時沒想清楚問題定義？
            - 還是修繕單問題定義本身漏了某個維度？
              ↓
            想清楚最佳解決方案 → 一次施工解決
              ↓
            第二輪 subagent 驗收
              ├─ PASS → 雙簽歸檔，完工（含結構分析紀錄）
              └─ FAIL → 🚨 系統性失效，拋 使用者 重新定義問題（極罕見）
```

**ping-pong 上限**：第二輪後必須結束。第二輪 FAIL = 系統性失效，必須由 使用者 介入。

#### 修繕單末尾雙簽段格式

施工 + 驗收完成後，工頭在原修繕單末尾追加：

````markdown
---

## 施工 + 驗收紀錄

**施工完成**：YYYY-MM-DD HH:MM 台北
**工頭證據**：
- 改動清單：N 處 Edit / N 個檔案（具體檔案路徑 + 行範圍）
- grep 驗證：[具體 grep 結果引用]
- mtime 驗證：[ls -la 結果]
- 結構驗證：[預期結構就位]
- 反向隔離：[未動其他系統]

**驗收方**：{對方負責人 subagent 名稱 / 使用者 親簽}
**驗收結果**：✅ PASS（第一輪 / 第二輪含結構分析）
**驗收證據**：[對方負責人具體驗收項目 + 結果]

**雙簽**：工頭 fal ⚒️ + {對方負責人}
**結案歸檔**：YYYY-MM-DD HH:MM
````

## 🔴 M08 強制 SOP：核准型動作流程定義（2026-04-26 #037 升格）

> **來源**：賽事分析 專案 #036 v2 + #037 雙單合併。M08「核准 ≠ 寫入」連續實證 N+1 次後升格為**跨客戶通用 SOP**。

### 通用流程閘門

任何「核准型動作」（使用者拍板 / CEO 核准 / 工頭批准修繕單 / 任何「拍板後需要寫入」的流程）的流程定義必須結束於：

```
□ 核准結束（拍板完成）
□ 🔴 寫入完成（具體檔案 + 行範圍 / paste 內容）
□ 🔴 mtime 驗證為當日（ls -la 看 date）
□ 🔴 結構驗證（grep 預期結構就位）
□ 🔴 結案歸檔（修繕單 mv 到 archived/、工作簿登記完工、看板更新）
```

**禁止**「核准 → 結案」的省略寫法。中間缺一步 = 「核准 ≠ 寫入」必復發。

### 適用範圍

- 修繕單施工（使用者 拍板 → 工頭施工 → grep 驗證 → 驗收紀錄 + 歸檔）
- pattern-log 建卡（CEO 核准 → 覆盤 Skill 寫入 → mtime 驗證 → 修繕單歸檔，賽事分析 專屬）
- schema 必填欄位升級（規格定稿 → 升級時間戳常數 + 判定分支 + 表格 append + 寫入硬規則四件套；S-INFRA-26 教訓條目）
- 任何「使用者授權後工頭代為施工」的流程

### 配套機制

- **F-004 即時驗收雙簽協議**（2026-04-26 上線）：工頭施工完當下召喚對方負責人 subagent 驗收 + 雙簽 + 即時歸檔，三系統同步——從根因封住「核准 ≠ 寫入」隱性鉤子（不依賴下次對話會主動讀修繕單）
- **#038「掃 pending 修繕單」鉤子退役**（2026-04-29 F-011）：F-004 即時雙簽 + 客戶開單立刻修工作流自然封住根因，#038 上線後未實證 helpful → 拿掉避免雜訊
