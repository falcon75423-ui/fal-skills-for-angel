# 規則對策層級

> 讀取時機：SKILL.md 世界觀條目「規則層的對策有層次性」延伸詳述。
>
> 來源：F-016（2026-04-30）淬鍊三輪 R1 → R2 → R3 衝撞驗證後定型。M01「規則進入動線 = 假安全感」（helpful=7）升級為 SKILL.md 世界觀的工法載體。

---

## 對策層級對照表

| 層級 | 對策 | 適用場景 |
|------|------|---------|
| **0** | **消除/重定義（重新設計問題）** | **對策方向錯 / 規則本身有問題（永遠的退回選項）** |
| 1 | 人腦記得掃 | 一次性 / 罕見 |
| 2 | 規則進入動線（M01）| LLM 紀律可遵守 |
| 3 | 升 Hook 層攔截 | LLM 物理失效（reasoning 構造 string literal 失真等）|
| 4 | 腳本主動掃（dogfood validator）| 累積追蹤類自驗（M35 / S-INFRA-29）|
| 5 | CI 自動驗證 | 高頻自動觸發 |

🔴 **判別規則**：對策失效時**不在同層補規則**——必須升上一層（強度不夠）或回層級 0 重新設計（方向錯）。同層加強 = 該層已失效病理訊號。

對應 使用者 開發憲法第八條（升級順序：消除 > 自動化（Hook）> 工作流閘門 > 記憶）。

---

## 🔴 升級前置檢驗（M26 根因問句，必附過閘證據）

對策 N 失效時，**升級前必先回答**：

| 分支 | 失效原因 | 對策 | 舉證要求 |
|------|---------|------|---------|
| **(a)** | **強度不夠** | 升上一層（N → N+1）| ≥ N 次失效紀錄（可驗證）|
| **(b)** | **方向錯**（規則本身有問題）| 回層級 0 重新設計 | 必附**新規則初稿 + 砍規則退什麼**（一進一出對齊）|
| **(c)** | **失效成因不純**（環境/噪音）| 降為 N=1 觀察期 | 環境問題的具體證據（如 antivirus 隔離 / cross-platform 差異）|

🔴 **複合失效拆解 SOP**：多分支同時成立時，**先濾 (c) 排除噪音 → 確認非噪音後再分 (a)/(b)**。

🔴 **過閘證據**：判定 + 證據必須出現在**升級提交物**裡。缺證據 = 視同沒過 M26（跟拍板骨架同層形式硬抓手）。

---

## 升級提交物（多載體）

「過閘證據」不限「拍板訊息」單一載體。任一合法：

| 載體 | 適用情境 |
|------|---------|
| 拍板訊息「為什麼」段 | 使用者 + 工頭對話內升級 |
| subagent log | subagent 自主升級回報 |
| commit message | 程式碼層升級 |
| 修繕單 | 跨對話 / 跨 Skill 升級 |
| Bridge Document | 鍛造主體升級 |

🔴 紀律：**所有載體都必須含「分支判定 + 證據」**——換載體不換實質。

---

## 同層改寫 vs 同層加強（字面邊界）

「同層加強 = 病理訊號」**不適用**於以下情境：

| 動作 | 是否「同層加強」？ | 原因 |
|------|------------------|------|
| 規則本身**重寫**（不加新機制）| ❌ 不是 | 算「同層改寫」，需走 M26 + 一進一出 |
| 規則**擴觸發條件** | ✅ 是病理 | 補丁式擴條件，禁止 |
| 規則**加新檢查項目**（不退舊機制）| ✅ 是病理 | 違反一進一出 |
| 升級前**寫新規則初稿**（為了走分支 (b) 回層級 0）| ❌ 不是 | 屬升級舉證的一部分 |

🔴 邊界判定：「重寫 vs 加強」分歧時，看是否**有新機制誕生**。改寫舊機制邏輯 = 重寫；保留舊機制 + 加新層 = 加強。

## 🔴 作用域宣告（meta 層豁免）

**本工法僅適用「運行期規則失效」**——SKILL.md 世界觀條文 / references / lessons 條目本身的設計演化**不適用**。

**為什麼**：
- 世界觀條文的演化**本質是設計過程**，不是運行期執行（如本檔 R1 → R2 → R3 三輪迭代每次都「重寫條文」屬正常設計流程）
- 把「同層加強 = 病理」套到設計過程 = 邏輯自爆（淬鍊系統用補丁修補「禁止補丁」的條文）
- 設計過程有自己的方法論：淬鍊師三情境衝撞 + 收斂標準 + 四輪上限

**判別**：
- 運行期：規則已上線、實作中遇失效 → 適用本工法
- 設計期：規則尚未上線、迭代設計中 → 不適用（走淬鍊師工法）

## 升級訊號通用門檻

| 層級 | 升級門檻 |
|------|---------|
| 1 → 2 | 同類錯誤 ≥ 3 次（Three Sightings） |
| 2 → 3 | 進入動線後仍復發 ≥ 2 次 |
| 3 → 4 | Hook 啟用後仍失效 ≥ 3 次（30 天內） |
| 4 → 5 | 沒人主動跑腳本（觀察期） |
| 5 失效 | 回層級 0 重新設計（不存在第 6 層）|

🔴 **每個門檻達標時都要過 M26 三分支判定**——不允許「達標 = 自動升級」。

## 案例庫

### F-007 → F-009/F-010：對策 2 → 對策 3 升級鏈

**情境**：修改三關（讀關）規則
- F-007（2026-04-27）：規則進入動線（對策 2 上線）
- F-009（2026-04-28）：進入動線後仍 4 次連續同根因 Edit fail
- M26 三分支判定：(a) 強度不夠（規則對 LLM 紀律可遵守，但 reasoning 構造 string literal 物理失敗）→ 升對策 3
- F-010：升 Hook 層攔截（PreToolUse Hook 偵測 codepoint > 0x007F）

### F-013：對策 3 內部精煉

**情境**：F-010 Hook 上線後 #040 + F-011 兩次失效
- M26 三分支判定：(c) 失效成因不純（Hook stderr 物理上不傳給 LLM 是訊號傳遞層問題，非邏輯失效）
- 處理：對策 3 同層**改寫**（不加新層）→ 訊號傳遞改用 JSON `hookSpecificOutput.additionalContext`
- 同對話 E2E 驗證 LLM 收到警示

### F-016：對策 4 上線

**情境**：飛輪規則「同類錯誤 ≥ 3 次升級閘門」靠人腦記得掃
- 升級理由：M01 自身啟示——規則進入動線（對策 2）對「累積追蹤類規格」不夠
- 處理：寫 dogfood-validator.py 主動掃自家飛輪規則
- 對策 4 上線首跑就抓自家 v1 兩個設計缺陷（M35 dogfood 自證有效）

## 配套機制

### 偵測（下游）

- **踩雷錄「閘門失效追蹤」表**：「升級」欄記每次對策升級事件、「狀態」欄記當前對策層級
- **F-016 dogfood validator A check**：抓「累積追蹤表次數 ≥ N 但狀態未含結構性對策」
- **「對策升級失敗」訊號**：對策 N 失效後若工頭/使用者反射動作是「在 N 加強規則」→ 病理訊號

### 退役機制（一進一出）

對策 N 升級到對策 N+1 後：
- 對策 N 的舊規則**保留**作為 fallback（如 F-010 Hook 上線後紀律層 verbatim 複製仍是 fallback）
- 但對策 N 的「擴條件 / 補丁式加強」歷史紀錄需在踩雷錄標記「已升級」狀態（防 dogfood 誤報）
