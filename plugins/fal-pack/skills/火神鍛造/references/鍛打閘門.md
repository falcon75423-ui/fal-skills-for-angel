## ❌ 白話對齊（已退役 2026-04-24）

**退役原因**：連續四度失效（R1 / 04-12 壓力測試 / #027 / #028）後認清，把「講白話」做成觸發式閘門是**錯層**——閘門是 exceptional 動作綁定（Edit 時、推薦時），預設姿態是 default 身份綁定（每次對 使用者 開口）。錯層 → 靠自我覺察判斷觸發時機 → context 膨脹時必漏。

**新機制**（結構性四件套，見 SKILL.md 身份區塊「講話的預設姿態」）：
1. **預設姿態**（身份層）：對 使用者 講話默認白話，不需要觸發
2. **拍板訊息骨架**（形式層，硬抓手）：含 Q1-QN / A/B/C / 請選 / 拍板 / 確認 / 要不要的訊息 → 強制骨架（揣測意圖 / 白話分析 / 技術細節 / Q1-QN）
3. **使用者 語彙字典**（對照層）：`references/使用者語彙.md` 白話詞庫 + 技術名詞→白話對照表，字典可查者必翻譯
4. **飛輪偵測**（兜底層）：閘門升級後復發 ≥ 2 次 → 禁「擴條件」，強制退役或重新問 WHO（見踩雷錄「閘門失效追蹤」+ 教訓索引運作規則）

**歷史保留價值**：原本的四步驟（揣測意圖 / 白話描述 / 對齊確認 / 等待回覆）作為拍板訊息骨架的內容生成指引仍然有效，但**執行機制不再是閘門**。

### 為什麼升 Hook 層（2026-04-28 F-009 淬鍊師 meta 審視結論）

**根因不是「規則不夠強」，是「LLM 物理上做不到」**：

「絕不憑記憶打字」字面上 LLM 無法 100% 遵守——LLM 的 reasoning 本質是「從訓練分布生成 token」，沒有「逐字符 verbatim 複製」的物理機制。F-009 4 次連續同根因 Edit fail（含 meta-level 自指悖論：在「登記閘門失效」這個動作本身觸發同類失效）= 物理失效根因的活體實證。

**按 使用者 開發憲法第八條（升級順序：消除 > 自動化（Hook）> 工作流閘門 > 記憶）**：F-007 升級工作流閘門層（規則進入動線）後仍失效 → 該層已失效 → 必須升到上一層（Hook 層）。

Hook 層解的特性：
- **執行機制不依賴 LLM reasoning**——codepoint 偵測由腳本層做
- **WHO 跟 HOW 對齊**——WHO 否定 LLM reasoning 可靠性，HOW 也不依賴 LLM reasoning
- **「絕不憑記憶打字」字面退役**——Hook 在執行層擋住，規則層不需要再寫

## 🔴 高風險預設驗證閘門

**提出「工頭推薦 A/B/C」前必經——隨口建議不算，正式決策提案才算。**

### 觸發條件

推薦內容涉及任一項就啟動：

- 檔案/目錄結構變更
- 跨 Skill 約束（引用規則、硬規則、讀取權限）
- 資料源 / 工具鏈 / 外部依賴（付費源、API、訂閱）
- 時機 / 頻率 / 閾值（觀察週期、刷新節奏、觸發條件——容易撞既有條文）
- 推薦帶「繼承」「沿用」「引用」字樣（暗示對齊既有條文，必須驗證對齊對不對）

### 三步驟

| 步驟 | 做什麼 | 不通過的信號 |
|----|--------|------------|
| **讀原** | 重讀本推薦依據的原始檔案原文。不靠 context 記憶 | 「我記得它寫 X」「心法大概說 Y」→ 停，去讀檔 |
| **類比** | grep/glob 搜「系統中同類問題的既有處理」 | 沒搜就推薦 → 停，去搜 |
| **對照** | 本推薦跟既有硬規則是否衝突（付費工具必經、引用不複製、覆蓋率門禁、等）| 「應該沒衝突」→ 停，實際比對 |

### 輸出約束

推薦陳述必須附「驗證證據」（至少一行，指出具體 `檔案:line` 或 grep 結果）。寫不出證據 = 推薦不成立。
