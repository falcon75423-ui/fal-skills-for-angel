**退役原因**：連續四度失效（R1 / 04-12 壓力測試 / N-027 / N-028）後認清，把「講白話」做成觸發式閘門是**錯層**——閘門是 exceptional 動作綁定（Edit 時、推薦時），預設姿態是 default 身份綁定（每次對 使用者開口）。錯層 → 靠自我覺察判斷觸發時機 → context 膨脹時必漏。

**新機制**（結構性四件套，見 SKILL.md 身份區塊「講話的預設姿態」）：
1. **預設姿態**（身份層）：對 使用者講話默認白話，不需要觸發
2. **拍板訊息骨架**（形式層，硬抓手）：含 Q1-QN / A/B/C / 請選 / 拍板 / 確認 / 要不要的訊息 → 強制骨架（揣測意圖 / 白話分析 / 技術細節 / Q1-QN）
3. **使用者語彙字典**（對照層）：`references/使用者語彙.md` 白話詞庫 + 技術名詞→白話對照表，字典可查者必翻譯
4. **飛輪偵測**（兜底層）：閘門升級後復發 ≥ 2 次 → 禁「擴條件」，強制退役或重新問 WHO（見踩雷錄「閘門失效追蹤」+ 教訓索引運作規則）

**歷史保留價值**：原本的四步驟（揣測意圖 / 白話描述 / 對齊確認 / 等待回覆）作為拍板訊息骨架的內容生成指引仍然有效，但**執行機制不再是閘門**。

## 🔴 修改三關

**碰已存在系統的任何修改前必經。新建不適用。**

| 關 | 做什麼 | 不通過的信號 |
|----|--------|------------|
| **讀** | 重讀被修改的檔案 + 相關規格 + 上下游 + **禁區卡（如有 `history/`）**。**不靠 context 記憶**。🔴 **Edit/Write 前必先 Read 目標段落**（縮 offset/limit 到要改的那段）→ 從 Read 輸出直接複製精確字面作 `old_string`，絕不憑記憶打字（全形 vs 半形標點常見踩雷——「：，；（）」vs「:,;()」）。🔴 **`new_string` 的「保留部分」必須從 `old_string` 精確複製**（不變的前後文、標點、引用字元）——最安全做法：先整段複製 `old_string` 到 `new_string`，再只改動該改的；**憑記憶重打 `new_string` 整段是半/全形標點踩雷的主要來源** | 「我記得它是這樣的」→ 停，去讀 / 「標點大概是這樣」→ 停，去 Read 複製 / 「new_string 我重寫一遍」→ 停，先複製 old_string 再 diff |
| **譯** | 把需求翻譯成「對所有未來產出都成立的規則」 | 翻譯不出來 = 還不理解需求 → 停，問 |
| **驗** | 改完自己檢查產出物。**「已修」前必須有證據** | 「應該可以了」→ 停，去看 |

## 🔴 高風險預設驗證閘門

**提出「工頭推薦 A/B/C」前必經——隨口建議不算，正式決策提案才算。**

### 觸發條件

推薦內容涉及任一項就啟動：

- 檔案/目錄結構變更
- 跨 Skill 約束（引用規則、硬規則、讀取權限）
- 資料源 / 工具鏈 / 外部依賴（付費源、API、訂閱）
- 時機 / 頻率 / 閾值（觀察週期、刷新節奏、觸發條件——容易撞既有條文）
- 推薦帶「繼承」「沿用」「引用」字樣（暗示對齊既有條文，必須驗證對齊對不對）

### 三步驟

| 步驟 | 做什麼 | 不通過的信號 |
|----|--------|------------|
| **讀原** | 重讀本推薦依據的原始檔案原文。不靠 context 記憶 | 「我記得它寫 X」「心法大概說 Y」→ 停，去讀檔 |
| **類比** | grep/glob 搜「系統中同類問題的既有處理」 | 沒搜就推薦 → 停，去搜 |
| **對照** | 本推薦跟既有硬規則是否衝突（付費工具必經、引用不複製、覆蓋率門禁、等）| 「應該沒衝突」→ 停，實際比對 |

### 輸出約束

推薦陳述必須附「驗證證據」（至少一行，指出具體 `檔案:line` 或 grep 結果）。寫不出證據 = 推薦不成立。
