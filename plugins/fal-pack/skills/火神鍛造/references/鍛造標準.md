# 鍛造標準（建築法規）

> 讀取時機：鍛打階段，準備產出結構時載入。或進行「驗收」時作為品質對照。

工頭不蓋違建——但建築法規服務建築，不是建築服務法規。工頭有權判斷某個鍛造物不需要某些結構。**判斷依據是靈魂，不是清單。**

---

## 完整 Skill 結構

不是每個鍛造物都需要全部——輕量工具、prompt template、功能模組各有各的適當形式：

- [ ] **Tradeoff 宣告（Skill 開頭高注意力區）**：一句話明文揭露「這套 Skill 的代價與適用範圍」——偏向哪一側（速度/謹慎/深度/覆蓋），瑣碎任務可否繞過
  - 功能一：給工頭自己明確的「可繞 vs 不可繞」判斷錨點，避免判斷漂移
  - 功能二：給使用者正當逃生口（瑣碎案例不走全套），保留部分採用而非全有全無
  - 功能三：擋住自己未來的「規則通膨」——每次想加新規則時會被自己那句話打臉
  - 格式範例：「{Skill 名稱}偏{取捨方向}。{什麼情境可繞}。代價是 {X}；收益是 {Y}」
  - 來源：karpathy-skills `CLAUDE.md` 開頭「Tradeoff: These guidelines bias toward caution over speed. For trivial tasks, use judgment.」
- [ ] **身份（WHO）**：內部按穩定度分層（最不變的在最上面）
  - [ ] **靈魂**（存在目的）— 幾乎不變。所有鍛造物必備
  - [ ] **心**（核心信念）— 緩慢演化。標準以上需要
  - [ ] **腦**（決策原則）— 隨經驗調整。重型需要
  - [ ] **角色呈現** — 有畫面感的隱喻 + 世界觀 + 人格張力
  - 🔴 每層存在就必須寫到同等精細度（規格密度均勻化）
- [ ] **人格**：用張力定義，不是特徵清單
- [ ] **觸發條件**：多層召喚詞（崗位名稱 / 完整綽號 / 關鍵字）+ 意圖觸發
- [ ] **思考姿態（HOW）**：2-3 個認知模式，用領域隱喻命名
- [ ] **路徑選擇**：由情境條件決定，判斷標準明確
- [ ] **品質校準**：「代表誰」決定標準，有底線
- [ ] **盲點**：寫進身份，是功能不是缺陷
- [ ] **成長機制**：session template + 回寫路徑

## 按需結構（由建築需求決定）

- [ ] 跨 Skill 握手協議（Brief 格式）
- [ ] **資訊架構設計（Progressive Disclosure + 注意力權重配置）**：Skill 是資料夾，不是一個檔案。設計三層載入結構：
  - **L1 Metadata**（name + description）— 永遠在 context 裡（~100 字），決定觸發
  - **L2 _index.md body** — 觸發時才載入（🔴 **≤ 300 行預算**），核心流程與指引。每一行都要通過「刪掉它會不會斷」的測試，不會斷的搬到 L3 或刪掉
  - **L3 支援檔案** — 需要時才讀（`references/` 知識文件、`scripts/` 可執行腳本、`assets/` 模板素材），無上限。scripts 可以不讀進 context 直接執行
  - 在 _index.md 中明確列出每份支援檔案的用途與讀取時機，讓模型自主決定何時深入
  - **🔴 _index.md 內部注意力權重配置**（context 的注意力開頭和結尾最強，中間最弱）：
    - **高注意力區（開頭）**：靈魂 / 身份 / 世界觀 / 觸發條件 — 最穩定、最重要、定調用
    - **中注意力區（中段）**：思考姿態、路徑選擇、品質底線 — 放簡版 + L3 指針，不放細節。中段薄，注意力流失影響最小
    - **高注意力區（結尾）**：必經齒輪清單 🔴 / 盲點 / 互動指令 — 最容易被跳過的放這裡
- [ ] 工具配置（CLI、MCP、API 等）
- [ ] 互動指令（用領域語言）
- [ ] **`landmines.md`（已知地雷）** — 有工具操作的 Skill 必備。Skill 的免疫系統：每踩一次雷就自動免疫。格式：`❌ 地雷 → ✅ 正確做法`。_index.md 寫入強制載入指令

## 🔴 WHAT 層施工規範（Skill 涉及代碼產出物時必讀）

> **教訓來源**：賽事分析 預測公司 0322 修繕——7 輪修繕、20+ 份檔案，所有問題都出在 WHAT 層。工法在 WHO/HOW 很扎實，但代碼層幾乎空白。

當 Skill 的 WHAT 層涉及**腳本生成產出物**（Excel、PDF、HTML 等），以下五條必須在鍛造時寫進規格：

**1. 「應該」掃描**
> 規格中每一句「X 應該會 Y」都是未爆彈。掃描整份規格，把所有「應該」轉為閘門。
> - 「格式應該一致」→ 格式定義集中在一個模組，腳本引用
> - 「列高應該夠」→ `auto_row_height()` 函式按內容計算
> - 「覆盤應該同步」→ 更新腳本寫入三處，checklist 逐一勾

**2. 單一真相來源**
> 任何數據點只能有一個權威來源。多處定義 = 必然漂移。
> - 格式 → `design_system.py`（一處定義，所有腳本引用）
> - 狀態判斷 → 一個工件（如 `覆盤.md` 存在與否），不是三種邏輯
> - 欄位順序 → 規格文件定義一次，腳本遵守

**3. 生成 vs 更新 分離**
> 生成腳本（從零建）和更新腳本（改既有）是兩種完全不同的東西。更新腳本的額外要求：
> - 寫入必須帶格式（font/fill/alignment/border），不能只寫 value
> - 必須處理合併儲存格（unmerge → 寫入 → re-merge）
> - 冪等性：跑兩次不破壞數據
> - 寫入位置必須明確（不靠「找到空行」這種脆弱邏輯）

**4. 程式化交付驗證**
> Agent 生成的代碼「能跑」≠「品質對」。腳本末尾必須有自動驗證段：
> - 結構驗證：Sheet 存在、欄數正確、行數合理
> - 格式驗證：抽查 N 個 cell 的 font.name / fill / height 是否符合設計系統
> - 內容驗證：關鍵欄位非空、數值在合理範圍
> - 驗證不通過 → 報錯，不靜默交付

**5. 嵌入式腳本（Scripts 作為一等公民）**
> 如果每次執行這個 Skill 都會重寫類似的腳本（資料撈取、格式轉換、驗證邏輯等），就應該預寫好放進 `scripts/` 資料夾。讓 Claude 花時間在「組合與判斷」，而不是「重建輪子」。
> - 判斷標準：跑三次 Skill，如果三次都獨立寫了類似的 helper → 抽出來變成嵌入式腳本
> - 腳本可以不讀進 context 直接執行（token-efficient、deterministic）
> - 放在 `scripts/` 資料夾，_index.md 寫明何時呼叫、怎麼呼叫

**6. 簡版設計附升級信號**
> YAGNI 要求先寫簡版，但「什麼時候升級」不能靠感覺——必須明文寫出觸發信號。
> - 每個「先用簡版、現在不做複雜化」的設計決策後，**必須緊接一行升級信號**
> - 格式：「當 {具體信號} 出現時升級為 {複雜版方向}」
> - 範例：「單一函式處理折扣——當出現第二種折扣類型時升級為策略模式」「in-memory 去重——當跨程序共享狀態需求出現時升級為 Redis」
> - 為什麼：沒有明文觸發信號 = 每次遇到邊界情境都要臨場判斷 = 判斷會漂。信號寫在紙上，修繕時不用重新思考
> - 來源：karpathy-skills `EXAMPLES.md` 「When to add complexity: Only when you actually need multiple discount types.」

## 原則書寫格式：反例並排

> 來源：karpathy-skills `EXAMPLES.md` Anti-Patterns Summary 表格。

當 SKILL.md 要表達一條行為原則時，**不只寫「該做什麼」，並排展示「典型錯誤行為」**。

**格式**：三欄對照表——`原則` | `典型錯誤做法` | `修正`。或段落形式 `❌ 錯版` / `✅ 對版` 並排。

**為什麼這比單寫對版有效**：
- 單寫對版 → Claude 讀了覺得「這是要求」，但遇到灰色地帶不知道「哪種偏差算錯」
- 並排錯版 → Claude 有參考點，自動對照當前行為離哪一側更近
- 對比效應：眼科驗光「A 比較清楚還是 B 比較清楚」比單看視力表更有效

**適用時機**：品質要求明確、違反代價高、灰色地帶常見的原則。不是每條原則都要並排——顯然的不需要（避免注意力稀釋）。

## 品質判斷（適用於所有形式的鍛造物）

- 定義層級對嗎？→ 行為是否能自然湧現？
- 遇到新情境 → 是否不需要新規則就能應對？
- 盲點 → 是否誠實面對自己的局限？
- 整體 → **是活的還是死的？**
- 形式對嗎？→ 這個需求真的需要這個規模的結構嗎？
- **MUST/NEVER 體檢** → 數一下全文的 ALWAYS、NEVER、MUST。每一個都問：這裡給 why 比給硬規則更有效嗎？過多的全大寫指令是 railroading 的訊號——讓 Claude 理解原因，比命令它服從更持久

## 工具盤點

鍛打過程中自然觸發的環節，不是獨立步驟：

1. 這個 Skill 需要什麼能力是 fal 本身沒有的？
2. 有現成的嗎？→ 搜 MCP registry、CLI 工具、npm、GitHub、API
3. 現成的夠用嗎？→ 評估穩定度、契合度、有沒有更好的
4. 沒有？→ 能自己造嗎？成本多大？→ 造最小可用版本
5. 造不了？→ 標記為「需要外部資源」，誠實說，不裝懂

## 跳板 Skill / Launcher 規範

> 適用時機：鍛造有雙層結構的 Skill 時載入。

### 什麼是跳板 Skill

Skill 的完整定義住在專案目錄（內容跟著專案走），但需要被 Claude Code 自動偵測載入時，會出現雙層結構：

```
~/.claude/skills/{name}/SKILL.md    ← 快捷入口（Launcher）
{專案}/skills/{完整 Skill}.md        ← 實際 SOP 定義
```

**跳板 Skill** = Claude 從 Launcher 跳進完整 Skill 的雙層架構。

### 根本原則：單一真相源

雙層結構的結構性風險：**兩個地方寫同樣的事實必然漂移**。

跳板 Skill 的防護 = **Launcher 薄到不可能漂移**：

- Launcher 只含「啟動指令 + 跨 Skill 分流」
- 所有業務內容（召喚語路由、引數列表、業務狀態）住在完整 Skill 的召喚協議段
- 當完整 Skill 新增/修改引數，Launcher 不動

### Launcher 薄化模板

```markdown
# {Skill 名稱} — 快捷啟動入口

{一句話：這個 Skill 在做什麼}

## 啟動
1. 讀取完整 Skill：`{絕對路徑}`
2. 執行其中的啟動序列
3. 所有引數由完整 Skill 的召喚協議處理

## 跨 Skill 分流（如有）
{簡表：何時該用這個 vs 其他 Skill}
```

### 🚫 Launcher 禁區

Launcher 絕對不得寫入：

- ❌ **引數列表**（「`召喚 X 覆盤` → 進覆盤」這種表格）——引數路由是完整 Skill 的職責
- ❌ **業務狀態**（「Phase X 待鍛造」「尚未支援」「Phase 4-C 上線」）——業務階段會變，Launcher 不跟著變 = 必然漂移
- ❌ **路由邏輯**（「啟動序列 → 顯示作戰選單」）——這是完整 Skill 的啟動序列內容

**判準**：問「完整 Skill 升級時，這句話要不要跟著改？」要改 → 這句話該在完整 Skill，不在 Launcher。

### 鍛打檢查

鍛造有 Launcher 的 Skill 時，鍛打階段必做「Launcher 禁區檢查」——掃描 Launcher 內容，確認三類禁區為零。
