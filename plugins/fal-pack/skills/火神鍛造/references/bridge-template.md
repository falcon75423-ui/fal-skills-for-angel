# Bridge Document 模板

> 讀取時機：鍛造或翻修結束時，取代流水帳式 session template。
> 設計來源：工匠筆記本（記「改了什麼、為什麼」）+ M&M Conference（決策點重建）+ SRE Postmortem（三欄反思）+ SAMULE（三層教訓）+ ACE（投票計數器）

---

## 格式

```markdown
# Bridge Document — [Skill 名稱]

**日期**：YYYY-MM-DD
**路徑**：開工 / 翻修 / 驗收
**類型標籤**：[工具層 / 情報層 / 思辨層 / 輸出層 / 領域系統 / 基建層]

## 三個關鍵決策點

> 不記流水帳。只記需要判斷力的分岔路口。

1. **[決策描述]**
   - 選了：...
   - 放棄了：...
   - 當時的判斷依據：...

2. **[決策描述]**
   - 選了：...
   - 放棄了：...
   - 當時的判斷依據：...

3. **[決策描述]**
   - 選了：...
   - 放棄了：...
   - 當時的判斷依據：...

## 三欄反思

| 做對了什麼 | 做錯了什麼 | 哪裡走運了 |
|-----------|-----------|-----------|
| ... | ... | ... |

> 「哪裡走運了」最重要——碰巧對了但判斷依據不夠強的決定，下次不能假設會再走運。

## 給下一個工頭的話

- ...
```

## 寫作原則

- **寫給「下一個拿起錘子的全新 fal」**，不是寫給自己。假設讀者零 context
- **決策點是靈魂**：三個決策點的品質決定整份 Bridge Document 的價值。如果找不到三個需要判斷力的決策，說明這次鍛造太簡單，可以省略 Bridge Document 只寫教訓
- **「哪裡走運了」不能空白**：每次鍛造都有走運的地方。寫不出來 = 沒有認真反思
- **教訓要可行動**：「下次要注意 X」不是教訓。「遇到 Y 情境時，做 Z 而不是 W，因為...」才是教訓
- **淬火摘要是品質軌跡的錨點**：二十份累積起來，模式會浮現
- **未完事項寫入 `clients/pending_followups.md`，不嵌看板條目註記**（F-014 升級 2026-04-29）：Bridge 結尾「後續事項」段可寫概要 + 指向 PF-XXX；active 狀態以 PF 表為單一真相源
