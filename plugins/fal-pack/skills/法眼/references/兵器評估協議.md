# 兵器評估協議

> 跨 Skill 共用的工具品質評估框架。法眼用快篩模式出刊，使用者用深審模式做技術審計。
> 精髓吸收自 [Github-Quality](https://github.com/Sac-Y/Github-Quality) 三層計分制，並擴展為跨平台適用 + 法眼品味層。

---

## 核心理念

- **動量 > 存量** — 週增量比絕對星數重要，正在起飛的比已經到頂的有信號價值
- **三層交叉驗證** — 數字、工程表面、外部口碑，任何單一維度都能騙人，三條線交叉才可信
- **拿不到資料不是 0 分** — 缺資料 = baseline + low-confidence 標記，不懲罰資訊缺失
- **專案類型決定評分邏輯** — 不同類型不同標準，不能一把尺量所有東西

---

## 雙模式

| | 快篩模式 | 深審模式 |
|---|---|---|
| **誰用** | 法眼（日常出刊） | 使用者（技術審計） |
| **耗時** | 3-5 分鐘 / 個 | 20-40 分鐘 / 個 |
| **深度** | L1-L3 + 品味層 | L1-L4 + 安裝評估 |
| **產出** | 三分制評級 + 法眼點評 | 完整評估報告 |
| **適用** | 批量篩選 5-10 個候選 | 對 1-2 個精選做深挖 |

---

## 專案類型自動分類

**依序檢查**（命中即停）：

| 優先序 | 條件 | 類型 |
|--------|------|------|
| 1 | `src/`, `lib/`, `app/`, `packages/` 為主幹 | **Software** |
| 2 | `agents/`, `skills/`, `commands/`, `plugins/` 2+ 且 README 描述框架 | **Agent-Framework** |
| 3 | datasets, benchmarks, papers, evals 為核心 | **Research** |
| 4 | 策展清單，幾乎無原始碼 | **Resources** |
| 5 | 以上都不是 | **Software**（預設）|

**特殊旗標**：
- ⭐ < 50 → 小專案模式（L1 降權，L2 升權）
- `archived` → 標記警告，L1 不套 time decay
- `fork` → 排除 fork/star ratio

## L1：客觀指標

> 核心問句：這個工具的數字健不健康？

### GitHub 來源（有 repo 時）

| 指標 | 重點 | 快篩做法 | 深審做法 |
|------|------|---------|---------|
| **動量** | stars/month，不是絕對星數 | 看 GitHub 頁面估算 | 用 API 精算 |
| **Fork/Star 比** | 0.08-0.30 健康，< 0.03 書籤型，> 0.60 異常 | 快速看比例 | 精確計算 |
| **Issue 處理率** | 全歷史 close rate | 掃一眼 | GraphQL 精算 |
| **Commit 活躍度** | 180 天內 commit 數 + 密度 | 看最近 commit 日期 | 三階段計分 |
| **Release 頻率** | 12 個月內 release 數 | 看 release 頁 | 計算間隔分佈 |
| **Time Decay** | `max(0.60, e^(-days/365))` | 憑最後 commit 日期估 | 精確計算 |

**年輕 repo 折扣**：< 6 月的 repo，Star Growth 和 Commit Activity 打 0.8-0.9 折，防爆紅即死。

**成熟穩定專案救濟**：Mature/Stable 且 180-730 天沒 commit，但星數 ≥ 1000 或有 release → Commit 保底 45 分，不判死。

### 非 GitHub 來源（跨平台適配）

| 平台 | 動量替代 | 採用信號 | 活躍度 |
|------|---------|---------|--------|
| **npm / PyPI** | 週下載量趨勢 | 依賴者數量 | 最近發版日期 |
| **Smithery / mcp.so** | 安裝數成長 | 評分、連接數 | 最後更新 |
| **SkillsMP / ClaudeMarketplaces** | 安裝數、投票趨勢 | 評分 | 最後更新 |
| **獨立網站 / CLI** | 社群提及頻率 | 引用數 | commit/release |

## L2：工程表面

> 核心問句：寫這個的人認不認真？

| 信號 | 判斷方式 | 快篩 | 深審 |
|------|---------|------|------|
| **README 品質** | 有安裝說明？有 API 文件？有截圖/GIF？有 badge？ | 掃一眼 | 10 分制 checklist |
| **目錄結構** | 程式碼組織清晰嗎？有測試資料夾嗎？Root 乾淨嗎？ | 看 tree | 9 分制（依專案類型） |
| **CI/CD** | 有 GitHub Actions？有 lint？有 test step？ | 看 `.github/workflows/` 存在性 | 讀 yml 內容計分 |
| **安全姿態** | dependabot？SECURITY.md？CodeQL？ | 看存在性 | 3 分制 |
| **Commit Message** | 規範格式？有描述？一致性？ | 掃最近 5 條 | 評估最近 20 條 |

**信號所有權（防重複計分）**：
- `docs/` → 歸 README，不歸 Directory
- `Makefile` → 歸 CI/CD，不歸 Directory
- `tests/` → 歸 Directory，不歸 CI/CD

## L3：社群口碑

> 核心問句：外面的人怎麼說？

### Issue 智能（有 issue 時）

| 維度 | 看什麼 |
|------|--------|
| **Bug 重複確認風險** | 有多少 bug 被多人確認但仍未解決？ |
| **Feature 迭代信號** | Feature request 有沒有真正的來回討論？ |
| **維護者參與度** | maintainer 有沒有在 issue/PR 裡認真回覆？ |

### Twitter/X 內容品質

逐條分類，不是數人頭：

| 類別 | 分數區間 | 說明 |
|------|---------|------|
| `news` | 20-60 | 發布公告、搬運、趨勢評論 |
| `feedback` | 35-85 | 真實使用經驗、優缺點比較 |
| `technical` | 50-95 | 獨立技術分析、架構拆解 |

**關鍵規則**：
- 具體批評 ≠ 低分。有理有據的負面 feedback 是高資訊量
- Hype marketing 壓低分
- 不看 likes/RT 數

### Hacker News 加成（只加不減）

| 討論量 | 加分 |
|--------|-----|
| 0 | +0 |
| 1-2 則 | +1 |
| 3-9 則 | +3 |
| 10-29 則 | +5 |
| 30+ 則 | +8 |
| 多條高討論 / 上首頁 | +10 |

### 缺資料時的 baseline

| 情況 | 處理 |
|------|------|
| Twitter 搜不到 | baseline 55 |
| Issue 關閉 | 權重轉到 PR + Twitter |
| HN 沒提及 | +0，不扣分 |
| 樣本 < 5 條 | 標記 low-confidence |

## L4：架構深度（深審模式限定）

> 核心問句：打開來看，裡面真的行嗎？
> 工具：understand-anything

**只在深審模式啟動。** 法眼快篩不跑這一層。

| 維度 | understand-anything 能回答的 |
|------|---------------------------|
| **模組設計** | 知識圖譜的 layers — 分層清不清楚、耦合度高不高 |
| **程式碼複雜度** | 每個 node 的 complexity 標記（simple/moderate/complex） |
| **呼叫鏈** | edges 的 calls/imports/depends_on — 有沒有循環依賴 |
| **測試真實度** | 不只看 `tests/` 存在，看測試 node 實際覆蓋哪些業務邏輯 |
| **Error Handling** | 函式級別的錯誤處理是認真寫的還是空 catch |
| **框架偵測** | 自動辨識用了什麼框架，判斷是否合理 |

**深審流程**：
1. Clone repo 到本地
2. 跑 `/understand` 產生知識圖譜
3. 用 `/understand-chat` 針對關鍵問題詢問
4. 必要時用 `/understand-explain` 對核心模組做深度解釋
5. 綜合 L1-L3 + L4 出完整報告

## 品味層（法眼專屬，不計分）

> 機械計分之上的主觀判斷。只有法眼會用這一層。

| 維度 | 判斷問句 |
|------|---------|
| **痛點真實度** | 這個工具解決的問題，使用者或他的讀者會碰到嗎？ |
| **設計品味** | 不只是能用，是用起來聰明嗎？API 設計優雅嗎？ |
| **生態位** | 這個品類已經有什麼了？它比現有方案好在哪？ |
| **意外值** | 這是 使用者自己找不到的東西嗎？ |

## 快篩模式輸出（法眼用）

快篩不出完整報告，用三分制 + 一句話評語：

| 評級 | 意思 |
|------|------|
| 🔥 **驚豔** | 數字健康 + 工程認真 + 口碑正面 + 解決真痛點 |
| ✓ **值得試** | 多數維度及格，有亮點但也有但書 |
| 👀 **觀望** | 概念有趣但太早期，或某個維度明顯弱 |

**快篩 checklist**（法眼心裡跑一遍，不需要全部展開在版面上）：

```
□ 動量：週增量明顯？還是停滯？
□ 年齡：太年輕要打折
□ README：3 秒內能看懂這東西幹嘛？
□ 目錄：打開 repo 第一眼，結構清楚嗎？
□ CI：有沒有在跑測試？
□ 最後 commit：多久前？
□ 口碑：Twitter/HN 有人提嗎？說什麼？
□ 痛點：這解決的是真問題還是玩具問題？
□ 生態位：同類型最強的是誰？它有什麼不同？
```

## 深審模式輸出

完整報告，包含四層計分 + 安裝評估：

```markdown
## 🔍 兵器深審報告

### 📊 總評：{score}/100（{rating}）

**類型**：{Software/Agent-Framework/Research/Resources}
**Repo**：{owner}/{repo}
**評估時間**：{date}

---

### 🛡️ L0 安全掃描 — {CLEAN/MEDIUM/HIGH/CRITICAL}

**L-1 安全著陸**：{CLEAR — .gitattributes 無 filter、無可疑 submodule}

**閘門 A 憲法防護**：
- AI 設定檔：{是否存在 CLAUDE.md/.cursorrules 等，內容摘要}
- 隱藏指令：{零寬字元/HTML 註解/參考連結/RTL/SVG 掃描結果}
- 指令覆寫模式：{ignore instructions 等模式掃描結果}

**閘門 B 供應鏈**：
- CVE 漏洞：{Critical: N / High: N / Medium: N}
- Secret 洩露：{掃描結果}
- 授權風險：{授權類型，是否有 AGPL 等傳染性授權}
- Build script：{postinstall/setup.py 等檢查結果}
- 二進位檔案：{是否存在不合理的 .exe/.dll/.so}

**閘門 C 惡意碼**：
- Semgrep：{ERROR 級發現數量與摘要}
- 後門模式：{eval+decode、動態 import 等}
- 可疑外連：{hardcoded IP/異常 domain}
- CI/CD：{第三方 action 鎖定方式、權限範圍}

### L1 客觀指標 — {score}/100
- ⭐ 動量：{detail}
- 🍴 Fork/Star：{detail}
- 🐛 Issue 處理：{detail}
- 📝 Commit 活躍：{detail}
- 📦 Release：{detail}
- ⏰ Time Decay：{detail}

### L2 工程表面 — {score}/100
- 📘 README：{detail}
- 📁 結構：{detail}
- ⚙️ CI/CD：{detail}
- 🔒 安全：{detail}
- 💬 Commit Message：{detail}

### L3 社群口碑 — {score}/100
- 🐛 Issue 智能：{detail}
- 🐦 Twitter/X：{detail}
- 📰 HN 加成：{detail}

### L4 架構深度 — {assessment}
（understand-anything 知識圖譜分析結果）
- 模組分層：{assessment}
- 核心複雜度：{assessment}
- 測試覆蓋：{assessment}
- 依賴健康度：{assessment}
- 特別發現：{findings}

### 🎯 白話能力卡 — 這東西到底能幹嘛？

> 用生活譬喻 + 實際情境解釋核心功能，讓非技術使用者一看就懂。
> 不解釋技術原理，只說「你能拿它做什麼」和「像什麼」。

**它是什麼**：{一句話生活譬喻，例如「curl 之於 HTTP = mcpc 之於 MCP」}

**你能拿它做的事**：
| 情境 | 你會怎麼用 | 像什麼 |
|------|-----------|--------|
| {實際使用情境} | {具體指令或操作} | {日常生活譬喻} |
| ... | ... | ... |

**它不能做的事**：{明確列出邊界，避免期待落差}

### 💡 綜合判斷
**優點**：
- {point}

**問題**：
- {point}

**安裝可行性**：{assessment}
**客製化空間**：{assessment}
**建議**：{recommendation}
```

## 交棒單格式（法眼 → 深審流程）

當 使用者對巡兵器推薦的工具表達興趣時，法眼輸出以下 code block 供複製：

```markdown
## 🔍 兵器審計請求

**Repo**: {repo URL}
**一句話**：{這東西幹嘛的}
**法眼快篩**：
  - 動量：{star 數 + 週增量}
  - 表面：{README/結構/CI 一句話}
  - 口碑：{Twitter/HN 一句話}
  - 法眼說：{品味判斷一句話}
**深審清單**：
  - [ ] 跑巡兵器深審
  - [ ] understand-anything 架構圖譜
  - [ ] 安裝可行性評估
**特別想知道**：{使用者額外提的問題，沒有就刪掉這行}
```

## 降級規則總表

| 情況 | 處理 | 不做什麼 |
|------|------|---------|
| GitHub API 失敗 | 標記 unavailable，重分配權重 | 不猜、不給 0 |
| 非 GitHub 平台 | 用替代指標（見 L1 跨平台表） | 不跳過 L1 |
| Twitter 搜不到 | baseline 55 + low-confidence | 不給 0 |
| Issue 關閉 | 權重轉 PR + Commit | 不跳過 L1 issue 權重 |
| 星數 < 50 | L1 降到 25%，L2 升到 55% | 不用大專案標準審判 |
| Archived | 標警告，不套 time decay | 不自動判死 |
| Fork | 排除 fork/star ratio | 不重複計原作者功勞 |
| 年輕 repo < 6 月 | 動量和活躍度打折 | 不因為新就加分 |
| 樣本太小 | 標 low-confidence，用 baseline | 不懲罰資訊缺失 |
