# NotebookLM 建檔與報告規範

> 讀取時機：精選文章要存入 NotebookLM 時載入。

---

## 建檔架構

| 筆記本類型 | 命名 | 用途 |
|-----------|------|------|
| **週報筆記本** | 法眼精選 — YYYY-WXX | 本週精選集中存檔、生成報告/心智圖 |

精選文章和影片存入當週週報筆記本。不再為作者建立獨立筆記本（維護成本高、實際使用率低）。

**來源標題格式**（新增來源時必遵守）：
```
[YYYY-MM-DD] [類別] 文章標題
```
範例：`[2026-03-24] [AI工程] Context Engineering Best Practices`

**YouTube 影片標題格式**：
```
[YYYY-MM-DD] [影片] 頻道名 — 影片標題
```
範例：`[2026-03-20] [影片] No Priors — Karpathy on Code Agents and the Loopy Era`

**Reddit 帖子標題格式**：
```
[YYYY-MM-DD] [Reddit] r/subreddit — 帖子標題
```
範例：`[2026-04-05] [Reddit] r/MachineLearning — Why I switched from RAG to long context`

---

## 🔴 單源單報告（鎖死齒輪）

每一個精選來源新增到 NotebookLM 後，必須用工作室的「報告」功能為該**單一來源**生成一份自訂報告。

- 絕對不可多個來源混合生成一份報告——內容會混雜，品質崩塌
- 一個來源 → 一個報告，例外零容忍
- 語言：繁體中文（⚠️ 這是給 Claude 看的設定提醒；NotebookLM 看不到本檔，所以下方 prompt 必須**明確再寫一次**給 NotebookLM 看）
- 報告格式：自訂報告

---

## 報告生成 Prompt（骨架固定，可附加 user 補強）

🔴 **報告標題格式**：必須與來源標題一致，前方帶日期與分類。例：`[2026-03-24] [AI工程] Context Engineering Best Practices`

```
報告標題前方要有日期與分類，並與來源相同。

🔴 來源若非繁體中文（英文 / 日文 / 其他）：先完整翻譯內容到繁體中文台灣用語再做報告。輸出全程中文，不夾雜原文（標題除外）。

報告的方式像一個懂內行的人在跟你聊天，不是一份報告在跟你說教。
聊起天來有節奏、引導感、畫面感、邏輯清晰、提到術語時銜接嵌入式翻譯，讓聽者可以自然理解術語。

報告中每一段落主題內容結束時，舉一個專業淺顯易懂的生活譬喻範例，洞悉並道出這個段落主題精隨。

「報告語氣」不是「工程師 / 分析師的簡寫」，就是像我們平常在聊天的那個聲音。
寫的時候想像他坐在你對面。每一段文字、每一個欄位，無例外。

整份報告寫完後，從第一個字到最後一個字默念一遍。
每一段問自己：「如果我當面對讀者這樣說，他會覺得我在跟他聊天，還是在唸一份報告給他聽？」
只要聽起來沒有符合上述標準轉譯，只是照本宣科唸報告的，重寫。
```

## User 客製 Prompt 處理（補強，不覆蓋）

使用者請法眼出報告時，可能提供自己的 prompt（譬如「重點放在實作步驟」「對比歷史趨勢」「強調對台灣讀者的啟示」）。**處理規則**：

- **預設**：上面「報告生成 Prompt」骨架**永遠保留**
- **append 模式**：user prompt 接在骨架後面，**不覆蓋骨架的風格鐵律**
- **衝突時的優先序**：骨架的風格鐵律（聊天語氣 / 中文翻譯 / 嵌入式術語翻譯 / 譬喻範例）優先；user prompt 主要影響「內容方向、重點、視角」
- **明確告知 NotebookLM**：用 marker 分隔骨架和 user 補強

**法眼組合最終 prompt 的格式**：

```
[上方「報告生成 Prompt」骨架原文]

--- 以下是使用者本次補充的方向 ---
[user prompt 原文]
```

**使用者沒給 prompt 時**：直接用骨架，不加 marker 段。

**異常處理**：user prompt 若直接違反骨架的風格鐵律（譬如使用者說「不用譬喻直接列點」「全英文輸出」），法眼向使用者回報衝突並請拍板，是要照骨架還是 override，**不要靜默處理**。
