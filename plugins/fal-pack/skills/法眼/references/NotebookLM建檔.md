# NotebookLM 建檔與報告規範

> 讀取時機：精選文章要存入 NotebookLM 時載入。

---

## 建檔架構

| 筆記本類型 | 命名 | 用途 |
|-----------|------|------|
| **週報筆記本** | 法眼精選 — YYYY-WXX | 本週精選集中存檔、生成報告/心智圖 |

精選文章和影片存入當週週報筆記本。不再為作者建立獨立筆記本（維護成本高、實際使用率低）。

**來源標題格式**（新增來源時必遵守）：
```
[YYYY-MM-DD] [類別] 文章標題
```
範例：`[2026-03-24] [AI工程] Context Engineering Best Practices`

**YouTube 影片標題格式**：
```
[YYYY-MM-DD] [影片] 頻道名 — 影片標題
```
範例：`[2026-03-20] [影片] No Priors — Karpathy on Code Agents and the Loopy Era`

**Reddit 帖子標題格式**：
```
[YYYY-MM-DD] [Reddit] r/subreddit — 帖子標題
```
範例：`[2026-04-05] [Reddit] r/MachineLearning — Why I switched from RAG to long context`

---

## 🔴 單源單報告（鎖死齒輪）

每一個精選來源新增到 NotebookLM 後，必須用工作室的「報告」功能為該**單一來源**生成一份自訂報告。

- 絕對不可多個來源混合生成一份報告——內容會混雜，品質崩塌
- 一個來源 → 一個報告，例外零容忍
- 語言：繁體中文
- 報告格式：自訂報告

---

## 報告生成 Prompt（固定，每次使用）

🔴 **報告標題格式**：必須與來源標題一致，前方帶日期與分類。例：`[2026-03-24] [AI工程] Context Engineering Best Practices`

```
報告標題前方要有日期與分類，並與來源相同。

報告的方式像一個懂內行的人在跟你聊天，不是一份報告在跟你說教。
聊起天來有節奏、引導感、畫面感、邏輯清晰、提到術語時銜接嵌入式翻譯，讓聽者可以自然理解術語。

報告中每一段落主題內容結束時，舉一個專業淺顯易懂的生活譬作為喻範例，洞悉並道出這個段落主題精隨。

「報告語氣」不是「工程師/分析師的簡寫」，就是像我們平常在聊天的那個聲音。
寫的時候想像他坐在你對面。每一段文字，每一個欄位，無例外。

整份報告寫完後，從第一個字到最後一個字默念一遍。
每一段問自己：「如果我當面對讀者這樣說，他會覺得我在跟他聊天，還是在唸一份報告給他聽？」
只要聽起來像是沒有符合上述標準轉譯，只是照本宣科唸報告的，重寫。
```
