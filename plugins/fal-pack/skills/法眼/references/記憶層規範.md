# 記憶層規範

> 讀取時機：粗篩任何候選文章前必須已載入。初始上線日：2026-04-11。

---

## 為什麼要有記憶層

法眼的使命是「每天五分鐘讀到值得五十分鐘深讀的東西」。這個承諾意味著**每個版位都不能浪費**。

但法眼過去沒有記憶——每次召喚都從零開始，結果發生結構性失憶：同一篇文章被反覆選進精選報，佔掉版位擠掉本該上版的其他好文。這叫**騙版位**。

一個真正的總編輯不會忘記自己寫過什麼。他翻開過去兩週的刊物，腦裡自動帶著「我推過 X、Y、Z」的記憶去讀今天的新稿。讀到第一篇，心裡會響起：「等等，這跟我上週的 X 有關聯嗎？多了什麼？」——這不是額外步驟，是主編的本能。

記憶層的目的就是**把主編的本能變成可執行的步驟**。

---

## 核心原則

**記憶層不是去重演算法，是主編的腦子。**

區別在哪？

| 去重演算法 | 主編的腦子 |
|-----------|------------|
| 用欄位比對（URL、作者、relevance hash） | 讀內容摘要 + 用語意判斷 |
| 二元輸出（重複/不重複） | 多層輸出（燒 / 標記 / 過但要寫連接語 / 過） |
| 決策過程是黑盒子 | 每個決策都能寫成一段話解釋 |
| 攔住重複的方式：刪除 | 攔住重複的方式：**重複變成脈絡** |

法眼用的是**後者**。

## 四題問答（粗篩第 0 條）

每次遇到一篇候選文章，法眼必須**逐題回答**以下問題。任何一題答不出或答得牽強，直接燒。

### Q1. 讀者看完這篇，比他看過最近 30 天我推過的同類文章，**多知道什麼**？

**測試方式**：寫出**具體 3 點**。不能是籠統描述，要是讀者腦中會新長出的認知。

**通過範例**：
> 上週推了 Gemma 4 發布，今天看到 SemiAnalysis Gemma 4 實測。多知道什麼？
> 1. 實際跑完 benchmark 的硬數字（不是官方 claim）
> 2. Gemma 4 在 tool use 上打贏 Llama 4 的具體幅度
> 3. SemiAnalysis 對 Gemma 的功耗分析——這是第一篇量化功耗的
> → 三點都是新增量，**過**

**不通過範例**：
> 上週推了 Karpathy LLM Wiki 原文，今天看到 Medium 上某博客寫「Karpathy 的知識庫革命，你也該試試」。多知道什麼？
> 1. ⋯作者的個人感想？
> 2. 把原文用不同句型再說一遍？
> 3. ⋯⋯想不出第三點
> → 寫不出具體三點，**燒**

**邊界情況**：
- 「多知道一點」不算通過，要是**獨立、可命名、可被讀者複述**的新認知
- 如果「多知道的」只有翻譯（例如英文原文 → 中文轉述），大部分情況下這不夠當精選。除非該翻譯同時提供了**文化轉譯**或**台灣本地視角**這類非翻譯的加值

---

### Q2. 如果要推這篇，**連接語怎麼寫**？

**測試方式**：寫出**具體一句話**，把這篇和過去推過的那篇連成脈絡。這句話會**強制寫進點評**。

**通過範例**：
- 「三週前我們看到 Gemma 4 發布聲稱超越 Llama 4，這次 SemiAnalysis 跑完實測，有一個數字讓人意外——⋯⋯」
- 「上週法眼推了 Karpathy 的 LLM Wiki 範式給寫 code 的人，Fox 這篇替你翻譯成不寫 code 的人的八成可行版本。」
- 「Zvi 上週批判 Anthropic RSP v3 把具體承諾換成模糊承諾，這週他把同一套顯微鏡對準 OpenAI，結果更糟。」

**不通過範例**：
- 「這是另一篇關於 Karpathy LLM Wiki 的文章」——不是連接語，是複述
- 「這篇跟上週推的有關」——太空泛，沒有引入新認知

**寫連接語的原則**：
1. 必須**承認上週推過什麼**（讓讀者感覺到法眼有記性）
2. 必須**說出這次的新點在哪**（證明不是重複）
3. 語氣自然，像主編在對副手說話，不像系統在解釋規則
4. **一句話就夠**。不要寫一整段。

---

### Q3. 這週的版位預算，這篇值得**擠掉誰**？

**測試方式**：說出這週其他候選中被擠掉的那一篇的**名字**。不能說「不知道」、「之後再看」。

**通過範例**：
> 「Fox 的中文拆解過了 Q1 和 Q2，值得擠掉⋯⋯嗯，這週已經有 Karpathy 本人的原文在必讀，Lopopolo 在必讀，Zvi RSP 在必讀。Fox 的價值是中文化，但它會擠掉的是 `#11 的 Eight years of wanting`——那篇角度也好但不如 Fox 直接回應本週熱點。**擠它**。」
> → 說得出被擠掉的名字，**過**

**不通過範例**：
> 「這篇挺好的，應該可以上？」
> → 沒說誰被擠掉，**燒**

**邊界情況**：
- 版位上限大約 **8-12 篇**（取決於當週候選品質）
- 如果這篇的排名進不了前 10-12，即使過了 Q1 和 Q2，也要燒
- 如果這週候選少（冷清週），排名門檻自動降低

---

### Q4. 版面整體節奏承受得了嗎？（V2，暫不做）

> 保留欄位，V1 不啟用。V2 上線時會加入「本週主題分佈統計」，避免連續多週版面重點相同。

## picks-index.md 的格式定義

### 結構

```
# 法眼精選記憶索引 ｜ [主題]

> Metadata（最後更新、涵蓋期間、格式版本、用途）

## 📖 說明

## 🗓 [當週] — 詳細版

### 必讀 🔥 / 值得看 ✓ / 法眼特選 👁️ / 商業實戰 🏢

**#N. [標題]**
- 📎 URL
- 作者 ｜ 發布日期 ｜ 分類
- **讀者從這篇多知道了**：
  1. 具體點 1
  2. 具體點 2
  3. 具體點 3
- **法眼點評主旨句**：「一句話」
- **相關實體**：實體 1、實體 2、實體 3
- **本週連載**：（選填）與 #X 構成 ⋯⋯
- **本週並見候選**：（選填）壓下或未上版的相關候選
- **扣分記錄**：（選填）這篇有什麼不完美

## 🗃 [過去幾週] — 簡版

| 日期 | 分類 | 標題 |
|------|------|------|

## 🧹 衛生規則

## 📝 指紋格式版本記錄
```

### 必填欄位

- **URL**（精確比對用，最便宜最硬）
- **作者 + 日期 + 分類**
- **讀者從這篇多知道了**（**3 點具體增量**，這是 Q1 的對照基準）
- **法眼點評主旨句**（下次寫連接語時可引用「上週我說 X」）

### 選填欄位

- **相關實體**（人物/產品/事件關鍵詞，用來做字串預篩加速）
- **本週連載**（如果這篇和本週另一篇構成對讀關係）
- **本週並見候選**（記錄哪些候選被壓下來）
- **扣分記錄**（這篇有什麼不完美，下次類似情況避免）

### 禁忌

- **不要寫結構化 JSON / YAML** — 這是給主編讀的筆記，不是給資料庫的 schema
- **不要寫相似度分數** — 閾值會引入「0.85 為什麼不是 0.80」的哲學爭論
- **不要省略「多知道什麼」** — 省了這欄，Q1 測試就失效

## 連接語寫進點評的強制規則

### 觸發時機

通過 Q2 的候選（能寫出連接語的那些），**必須**在精選報的點評中顯性寫出連接語。

### 放置位置

連接語放在**點評的第一句或第二句**，不放在段末。讀者第一眼就感覺到「主編記得自己寫過什麼」。

### 範例

```
## 🔥 #4 ｜ 必讀

**SemiAnalysis — Gemma 4 Benchmark Implementation Deep Dive**
〈Gemma 4 基準測試實測深掘〉
✍️ Dylan Patel（SemiAnalysis）
📎 [連結]

核心論點：⋯

**法眼點評**：三週前我們看到 Gemma 4 發布聲稱超越 Llama 4，這次 SemiAnalysis
跑完實測，有一個數字讓人意外⋯⋯（開始展開這次的新洞察）
```

### 反例

```
**法眼點評**：這是一篇很好的 Gemma 4 benchmark 分析⋯⋯
（❌ 沒有連接語，看不出法眼記得上週推過 Gemma 4）
```

### 語氣拿捏

- **對**：「上週我們推了 X，這次⋯⋯」、「三週前看到 Y，這次補上了⋯⋯」、「X 在上週的必讀裡提過這個方向，這篇是它的下一集」
- **不對**：「如 2026-W15 所推薦的⋯⋯」（像系統寫的）、「延續之前的討論⋯⋯」（太空泛）、「我之前寫過類似的⋯⋯」（沒說是哪篇）

## Audit Trail（出刊末段的透明化區塊）

### 為什麼要有

記憶層如果是個黑盒子，它會變暴君——法眼自己決定什麼能上什麼不能上，使用者無從介入。

每次出刊末段必須列出「**今日燒掉清單**」，告訴使用者哪些候選被記憶層擋下、擋下的原因、觸發的舊精選是誰。**讓誤殺變得可見、可抗議、可調整**。

### 放置位置

在精選報的「📊 今日版面統計」之後、「🌬️ 聽風」之前，新增一個獨立區塊。

### 格式

```
═══════════════════════════════════

🗑️ 記憶層審計 — 今日被燒掉的候選

本次記憶層攔下 X 篇候選。以下是每一篇的攔截理由：

---

**被燒 #1**：[候選標題]
📎 [候選 URL]
攔截理由：**Q1 未通過**——讀者多知道什麼？寫不出具體 3 點。只是把
[觸發的舊精選] 用不同句型再說一遍。
觸發的舊精選：上週 W15 必讀 #1「Karpathy — LLM Wiki Pattern」
→ 如果你覺得這篇該上，回覆「覆蓋 #1」我會重新評估。

---

**被燒 #2**：[候選標題]
📎 [候選 URL]
攔截理由：**Q3 未通過**——Q1 和 Q2 都過了，但這週候選太強，
它只能排到第 13 位，進不了 8-12 的版位。
若改排本週版位：會擠掉 [本次上版的 #X]
→ 回覆「擠掉 #X」我會交換。

---

...（依此類推）

═══════════════════════════════════
```

### 使用者覆蓋機制

使用者可以回覆：

- **「覆蓋 #N」** — 強制把被燒的 #N 候選重新評估並上版
- **「擠掉 #X 換 #N」** — 用被燒的 #N 候選替換本次上版的 #X
- **「燒對了」** — 確認攔截正確，會寫入 landmines.md 作為未來的邊界案例

---

## 衛生策略

### 時間窗口

- **詳細版**：最近 4 週（約 30 天）
- **簡版**：第 5-12 週（約 31-90 天）
- **Archive**：超過 90 天移到 `memory/archive/picks-YYYY-MM.md`

### 觸發 archive 的時機

- 當詳細版累積超過 **4 週**時，最舊的一週降級為簡版
- 當簡版累積超過 **8 週**時，最舊的一週移到 archive
- Archive 不參與即時 Q1-Q3 比對，但保留可查（使用者可問「2025 年法眼推過 Karpathy 嗎？」）

### 去重自檢

每次出刊結束時，記憶層自檢：
- picks-index 裡有沒有重複條目（同 URL）
- 有沒有詳細版和簡版同時存在的條目
- 發現異常 → 警示使用者+ 寫入 `landmines.md`

## MVP 範圍（V1）

**做**：
- ✅ URL 精確比對（零成本的第一道門）
- ✅ 四題問答（Q1-Q3），Q4 保留
- ✅ 連接語強制寫入點評
- ✅ Audit trail 區塊
- ✅ 使用者覆蓋機制（「覆蓋 #N」「擠掉 #X 換 #N」）
- ✅ 衛生規則（30 天 / 90 天 / archive）

**不做**（V2+）：
- ❌ 相似度閾值 / embedding 比對
- ❌ NER 自動抽取實體（V1 用手動寫入的「相關實體」欄位）
- ❌ 事件標籤自動生成（V1 由法眼寫點評時順手打）
- ❌ 版面節奏分析（Q4）
- ❌ 認知層級分類學（「新聞型 / 範式型 / 實作型 / 反思型」自動標籤）
- ❌ 主線分支結構（把同主題的延伸文章折疊展示）

## 使用流程（給未來的法眼自己看）

### 巡稿前（啟動檢查）

1. 載入 `memory/picks-index.md`（**必跑**，不可省）
2. 掃一遍詳細版的「讀者多知道什麼」和「法眼點評主旨句」——**帶著這份記憶去讀新稿**
3. 記住本週主軸和連載脈絡

### 粗篩時（過眼 → 粗篩第 0 條）

1. 對每篇候選，**先跑 URL 精確比對**（最便宜）
2. URL 命中 → 直接燒，寫入 audit trail
3. URL 沒命中 → 跑 Q1-Q3 逐題問答
4. 任一題答不出 → 燒 + 寫入 audit trail
5. 全部通過 → 進入原本的粗篩五項 + 五維評估
6. 通過 Q2 的候選 → 標記「連接語必寫」並記下那句話

### 寫點評時

1. 有標記「連接語必寫」的候選 → 點評第一句或第二句強制寫出連接語
2. 沒有標記的候選 → 照原本寫點評

### 出刊末段

1. 正常輸出精選報
2. 「今日版面統計」之後新增「🗑️ 記憶層審計」區塊
3. 列出本次被燒掉的每一篇 + 攔截理由 + 觸發的舊精選

### 收工時（寫回記憶）

1. 把本輪精選的 10 篇寫入 `memory/picks-index.md` 的「當週詳細版」區塊
2. 把上週的詳細版降級（如果需要）
3. 更新「最後更新」時間戳
4. 檢查衛生規則（archive 觸發）

## 失敗模式與應對

### 失敗模式 1：誤殺（漏掉有價值的追蹤報導）

**症狀**：本該上的連載文章被燒掉

**檢測**：audit trail 裡的「燒掉清單」應定期回顧。使用者也可以回覆「覆蓋 #N」當場糾正。

**應對**：
- 短期：使用者手動覆蓋
- 中期：把這次的邊界案例寫入 `landmines.md`，標註「Q2 應該能通過，但法眼沒抓到連接語」
- 長期：V2 加入認知層級分類學，讓「新聞型」和「實作型」即使同主題也能並存

### 失敗模式 2：漏殺（重複文章還是溜進來）

**症狀**：同主題文章還是被推了兩次

**檢測**：每週出刊後手動掃 picks-index，看有沒有實體/主旨句重疊的

**應對**：
- 短期：立即燒掉較晚的那篇（反向刪除）
- 中期：改進 Q1 的「具體 3 點」檢驗標準——可能是法眼太寬容地接受了籠統的答案
- 長期：引入 embedding 相似度作為補強（但仍以四題問答為主）

### 失敗模式 3：記憶膨脹

**症狀**：picks-index.md 超過 1000 行，Q1 比對變慢

**檢測**：檔案字數監控

**應對**：
- 立即觸發 archive 衛生規則
- 把最舊的一個月移到 archive
- 如果 archive 後還是太大，檢查衛生規則本身是否需要調整
