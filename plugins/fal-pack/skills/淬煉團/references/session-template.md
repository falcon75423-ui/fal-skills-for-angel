# Session Template

> 讀取時機：召喚淬煉團跑一場碰撞時。模板採顧問團規範。
>
> 核心：**5 個角色不是平行檢查表，是會互相打斷、質疑、收斂的對話**。

---

## 完整 Session 結構（4 段式）

```
0. 開場白（淬鍊師交代議題）
1. 第一輪：5 角色各自表態
2. 第二輪：互相質疑
3. 第三輪：交鋒收斂 + 最終立場
4. 淬鍊師定錨（在 5 角色之外）
```

---

## 段 0：開場白

### 淬鍊師講

```
淬煉團到場。5 位請就位。

> 結構審視者：等我看一下整體拓撲。
> 邊界挑戰者：哼，又有什麼漂亮方案要我來戳破？
> 策略偵察者：（沉默）……等聽完再說。
> 動機追問者：使用者帶著什麼煩惱來？
> 簡化批判者：先告訴我，這個東西為什麼要存在。
```

### 議題交代格式

```
今天淬煉的議題：

【背景】
{2-3 句脈絡——為什麼會出現這個議題}

【雛形】
{完整貼上待淬鍊的智慧 / 條目 / 規則 / 結構}

【我的設計初衷】
{它要解決什麼問題、想擋什麼失效模式}

【具體要碰撞的問題】
{1-3 個張力點——例如「這個分類對嗎」「這條規則邊界在哪」「這個結構會不會 over-design」}
```

🔴 **議題交代紀律**：必須包含「我的設計初衷」，否則 5 角色會從零開始猜測，浪費迴圈。

## 段 1：第一輪 5 角色各自表態

### 規則
- 每人 ≤ 100 字
- 表態你的初步觀察
- 不需要呼應其他人（這是純獨立發言）
- 順序：結構審視者 → 邊界挑戰者 → 策略偵察者 → 動機追問者 → 簡化批判者

### 範例（待淬煉議題：「修繕的兩種情境並列分類」）

```
**結構審視者**：先看分類學——「對外 vs 對內」這個維度涵蓋所有情境嗎？我看到「跨客戶」案件不在這兩類內，分類有洞。

**邊界挑戰者**：「兩種情境」這詞本身就是地雷區。三種情境怎麼辦？四種怎麼辦？預留升級接口了嗎？

**策略偵察者**：（沉默）……等聽完再說。

**動機追問者**：使用者打開這份分類時，他在做什麼決策？沒這個分類他會卡在哪？

**簡化批判者**：兩種情境真的需要分類嗎？「修繕」就是「修繕」，加分類會不會過度結構化？
```

---

## 段 2：第二輪互相質疑

### 規則
- 每人 ≤ 200 字
- 必須**指名道姓質疑** ≥ 1 個其他角色
- 鼓勵打斷模式（動機追問者打斷技術討論）
- 鼓勵主軸辯證（結構審視者 vs 簡化批判者）

### 範例

```
**結構審視者**：簡化批判者，你說「修繕就是修繕」——那兩種情境的觸發點不同、證據要求不同，硬塞同一個流程會有 race condition。分類**不是裝飾，是承擔不同責任的結構**。

**簡化批判者**：結構審視者，你的論點承認了「責任不同」——那為什麼一定要叫「兩種情境」？叫「責任分流」更精準。「兩種情境」會誤導讀者以為是現象分類。

**邊界挑戰者**：等等——你們在討論名稱，但動機追問者剛問的「使用者打開時在做什麼決策」沒人回答。沒回答這題，命名都是猜的。

**動機追問者**：謝謝邊界挑戰者點到我。那我擴展——「使用者打開分類時」可能有兩種：(a) 進場時對齊（要快速判斷）/ (b) 設計時參考（要完整視角）。兩種需求對分類粒度要求不同。

**策略偵察者**：（終於開口）我聽完了。問題不在「兩種 vs N 種」，問題在「分類是規格還是參考」。如果是規格 = 必須完整覆蓋（簡化批判者錯）；如果是參考 = 80% 覆蓋夠用（結構審視者錯）。先回答這個。
```

🔴 **這段是 session 的核心**——沒有指名質疑 + 主軸辯證，就只是 5 個獨立檢查表。

---

## 段 3：第三輪交鋒收斂 + 最終立場

### 規則
- 每人 ≤ 150 字
- 表達**最終立場**（含「採納/否決/有條件採納」字眼）
- 必要時放棄前面的論點（誠實 > 死撐）

### 範例

```
**結構審視者**：策略偵察者點得對——先答「規格還是參考」。如果是規格，我堅持要完整分類；如果是參考，我接受「主流程 + 三個情境例子」格式。**有條件採納**。

**邊界挑戰者**：我支持策略偵察者的二分。如果是參考，我撤回「預留升級接口」的要求——參考性質本來就會持續長大。**有條件採納**。

**策略偵察者**：根據前面 4 角色發言，我推測這是「規格層」——因為它出現在 SKILL.md，而 SKILL.md 是規格。**所以結構審視者方向對**。但要加「分類學完整性」驗證。**採納，附條件**。

**動機追問者**：使用者打開時是「進場時對齊」場景多——所以需求偏向「規格層」+ 快速判斷。我同意策略偵察者的判定。**採納**。

**簡化批判者**：好，是規格層。那我撤回「叫修繕就好」的論點。但「兩種情境」改為「兩種責任類型」更精準。**有條件採納**。
```

## 段 4：淬鍊師定錨（5 角色之外）

### 規則
- 淬鍊師**不在 5 角色內辯論**——她整合
- 必須給「下一步行動」
- 不允許「5 角色都同意所以採納」這種偷懶整合

### 模板

```
**淬鍊師定錨**：
- 結構審視者點出 [X 主張]
- 邊界挑戰者證實 [Y 補充]
- 策略偵察者警告 [Z 隱藏問題]
- 動機追問者拉回 [A 受眾洞察]
- 簡化批判者建議砍 [B 多餘部分]

**整合判斷**：[1 句話，含核心決策]

**下一步行動**：
1. [具體動作 1]
2. [具體動作 2]
3. [具體動作 3]
```

### 範例（接續上面的範例）

```
**淬鍊師定錨**：
- 結構審視者點出「分類學需要完整性 + race condition 風險」
- 邊界挑戰者證實「沒升級接口會卡」
- 策略偵察者警告「規格 vs 參考二分是真正的根因」
- 動機追問者拉回「使用者進場時對齊 = 規格層需求」
- 簡化批判者建議砍「兩種情境 → 兩種責任類型」（用詞精準化）

**整合判斷**：這是規格層分類，不是參考層分類。命名改「兩種責任類型」+ 補完跨客戶情境。

**下一步行動**：
1. 修繕工法.md 把「兩種情境」改名為「兩種責任類型」
2. 補一條「跨客戶情境（情境 C）」處理
3. 新增 race condition 檢查（在情境 A 流程中）
4. 補 description 加「責任分流」字眼提升觸發精度
```

```
5 角色詳細人設：見 ${CLAUDE_PLUGIN_ROOT}/skills/淬煉團/references/五角色詳細人設.md（請先讀完再開始扮演）

待淬煉的議題：

【背景】
{脈絡}

【雛形】
{完整貼上待淬鍊內容}

【我的設計初衷】
{要解決什麼 / 想擋什麼}

【具體要碰撞的問題】
{1-3 個張力點}

請按以下節奏跑：

段 1（每人 ≤ 100 字）：5 角色各自表態
段 2（每人 ≤ 200 字）：互相質疑（必須指名道姓）
段 3（每人 ≤ 150 字）：最終立場（含「採納/否決/有條件採納」字眼）

格式：
**[角色名]**：[發言]

報告 ≤ 1500 字。

提醒：
- 避免過快共識——5 角色互相認同 = 沒衝撞 = 沒深度
- 動機追問者要敢打斷技術討論
- 結構審視者建構 + 簡化批判者壓縮 = 主軸辯證
- 策略偵察者前期可沉默，後段才開口
- **不需要「淬鍊師定錨」段**——定錨由我（淬鍊師）讀完報告後做
```

## 對話節奏紀律

🔴 **三條硬規則**：

1. **5 角色發言必標角色名**（無「**結構審視者**：」前綴 = 違規）
2. **第二輪必須指名質疑**（無「{角色名}，我不同意」字串 = 違規）
3. **淬鍊師不在 5 角色內**（淬鍊師若進場辯論 = 違反定錨權）

🔴 **三條觀察重點**：

1. **避免過快共識** — 第一輪 5 個角色都同意 = 沒衝撞 = 重跑
2. **角色不脫節** — 每個角色獨立發言不互相回應 = 5 個檢查表 = 重跑
3. **定錨權留淬鍊師** — 淬鍊師被某角色帶著走 = 變成 6 角色辯論 = 重跑

---

## Session 失敗模式 + 處理

| 失敗模式 | 訊號 | 處理 |
|---------|------|------|
| 過快共識 | 第一輪 5 角色都點頭 | 重新出 prompt 強調「敢碰撞」「找漏洞而非驗證」 |
| 角色脫節 | 5 個獨立發言不回應其他人 | 重新出 prompt 強調「指名質疑 ≥ 1 人」 |
| 動機追問者沒打斷 | 5 角色全程談技術 | 重新出 prompt 強制動機追問者第二輪必出聲 |
| 主軸辯證沒形成 | 結構審視者跟簡化批判者不衝突 | 在 prompt 加「結構審視者主張完整 / 簡化批判者主張砍 = 主軸辯證」 |
| 全 5 個都採納 | 都說「採納」沒分歧 | 重跑——多視角的價值不存在 |

---

## 工法的工法（meta 層）

session-template 本身——

- **R1 自審**：4 段式結構合理嗎？（開場 → 表態 → 質疑 → 收斂 + 定錨 = 標準辯論結構）
- **R2 自審**：邊緣 case 處理（過快共識 / 脫節 / 沒打斷 / 沒辯證 / 全採納 = 5 種失敗模式都列了）
- **R3 自審**：會不會被當作形式主義？（靠淬鍊師判斷力——失敗 session 重跑）
- **R4 自審**：誰會用這套模板？（淬鍊師寫 subagent prompt 時直接複製貼上 = 高頻使用）

**結論**：本模板 R1-R4 自審通過，可投入使用。
