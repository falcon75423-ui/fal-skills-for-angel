## 模版識別與適配

> 不同類型的文件有不同的「慣例」——讀者的預期、格式的規範、敘事的節奏。識別文件類型是排版設計的第一步。

### 識別流程

1. **線索收集**：
   - 使用者明確告知（「這是一份合約」）→ 直接定位
   - Brief 中的文件類型欄位 → 直接定位
   - 從內容推斷：關鍵詞、結構特徵、語言風格
   
2. **類型定位** → 載入對應的文件類型規範（`references/文件類型規範/`）

3. **適配判斷**：
   - 這份文件 100% 符合某個類型？→ 直接套用慣例
   - 混合型（比如「帶數據分析的商業企劃」）？→ 主類型 + 次類型，主類型的慣例優先
   - 全新類型？→ 從零設計，但可以借鑑最接近的類型

### 內建文件類型

| 類型 | 關鍵特徵 | 規範檔案 |
|------|---------|---------|
| 法律合約 | 條款編號、定義條款、雙方簽署 | `文件類型規範/法律合約.md` |
| 商業企劃 | 痛點→方案→效益→時程→預算 | `文件類型規範/商業企劃.md` |
| 學術論文 | 摘要→文獻回顧→方法→結果→討論 | `文件類型規範/學術��文.md` |
| 會議記錄 | 出席→議程→決議→待辦 | `文件類型規範/���議記錄.md` |
| 報告類 | 摘要→分析→結論→建議 | `文件類型規範/報告類.md` |

新類型會隨使用經驗持續擴充。

## 內容填充邏輯（起承轉合）

> 不同文件類型有不同的「骨架」。表弟不只會排版，還懂得內容該怎麼流動。

### 通用敘事結構

無論什麼文件類型，好的文件都遵循某種**張力曲線**：

1. **破題**：讀者為什麼要讀這份文件？（建立動機）
2. **鋪陳**：背景、現狀、問題定義（建立共識）
3. **展開**：分析、論證、方案（核心內容）
4. **收束**：結論、建議、下一步（行動呼召）

### 類型特化的骨架

**商業企劃書**：
```
1. 執行摘要（一頁說完全部）
2. 問題定義（Pain Points — 讀者要在這裡點頭）
3. 解決方案（Solutions — 讀者要在這裡眼睛亮）
4. 市場分析（為什麼現在？為什麼我們？）
5. 執行計畫（時程 + 里程碑）
6. 財務預測（投入 + 回報）
7. 團隊介紹
8. 附錄
```

**學術論文**：
```
1. 摘要（Abstract）
2. 引言（Introduction）— 問題 + 貢獻
3. 文獻回顧（Literature Review）
4. 方法論（Methodology）
5. 結果（Results）
6. 討論（Discussion）
7. 結論（Conclusion）
8. 參考文獻（References）
```

**個人化報告**（如占星師的個人盤）：
```
報告骨架由內容方（作者）定義在「內容結構契約」中。
表弟的任務是理解這個結構，然後設計最好的視覺呈現。
不改骨架，改皮膚。
```

## 自動化大綱生成

> 使用者丟碎片資訊過來，表弟幫他整理出有邏輯的層級結構。

### 流程

1. **收集碎片**：使用者提供的所有資訊（文字、筆記、口述、聊天紀錄）
2. **主題萃取**：識別碎片中的核心主題和子主題
3. **邏輯排序**：
   - 時間序（事件報告）
   - 因果序（分析報告）
   - 重要性序（建議報告）
   - 類型的慣例序（學術論文、商業企劃）
4. **層級構建**：
   - H1：文件標題
   - H2：主要章節（3-7 個為宜）
   - H3：子節（每個 H2 下 2-5 個為宜）
   - H4：細項（只在必要時使用）
5. **驗證**：
   - 層級不跳級
   - 每個層級的粒度一致
   - 沒有「孤兒」章節（只有一個子節的章節要合併）
6. **呈現**：以 Markdown 大綱呈現，使用者確認後進入排版

### 大綱設計原則

- **3-7 法則**：每個層級的子項數量在 3-7 之間最好理解
- **粒度一致**：同層級的項目應該在相似的抽象程度
- **平行結構**：同層級的標題用相似的語法結構
- **獨立完整**：每個子節拿出來都能獨立閱讀

## 深度理解與視覺重設計

> 表弟最核心的價值：讀懂一份雜亂的文件，理解作者真正想表達的，然後用最好的視覺方式呈現。

### 「排版」指令的思考流程

1. **讀稿**（不跳過）：
   - 這份文件的核心訊息是什麼？
   - 誰是讀者？他們在什麼情境下讀？（螢幕/列印/投影）
   - 文件的情感基調是什麼？（嚴肅/友善/緊急/學術）

2. **診斷現狀**：
   - 現在的排版哪裡妨礙了閱讀？
   - 視覺層次跟內容層次對不對應？
   - 有沒有「所有東西都看起來一樣重要」的問題？

3. **設計方案**（與作者對話）：
   - 提出 2-3 個呈現方向，說明取捨
   - 用具體的視覺語言描述：「標題用 18pt 深藍，段落間距加大到 12pt，讓每個章節之間有呼吸感」
   - 聽作者的回饋，共同決議

4. **執行**：
   - 定義 Design System
   - 選擇引擎
   - 排版 → 驗證 → 交付

### 視覺重設計的核心判斷

| 判斷 | 問自己 |
|------|--------|
| 視覺層次 | 讀者的視線會按照我設計的順序移動嗎？ |
| 呼吸感 | 密集的資訊段落之間有足夠的留白嗎？ |
| 強調節制 | 粗體/色彩/底線的使用是否克制？太多強調 = 沒有強調 |
| 一致性 | 同層級的元素看起來一樣嗎？ |
| 功能性 | 表格比文字更清楚嗎？項目符號比連續段落更好讀嗎？ |

## 翻譯與本地化

> 文件級翻譯不只是翻文字，是翻一整本書。格式、術語、結構都要跟著走。

### 翻譯流程

1. **讀稿**：理解原文的意圖、語氣、讀者
2. **術語表建立**：提取專有名詞，確定翻譯方案（使用者確認）
3. **逐段翻譯**：
   - 保持原文結構（標題對標題、段落對段落）
   - 不直譯——用目標語言最自然的表達方式
   - 中文翻英文：拆長句、增主詞
   - 英文翻中文：合短句、減代名詞
4. **術語一致性掃描**：跑術語一致性引擎確認
5. **格式保留**：翻譯完成後，所有格式（樣式、字型、顏色、間距）保持不變
6. **校對**：四重閘門跑目標語言

### 本地化注意

| 面向 | 中→英 | 英→中 |
|------|-------|-------|
| 日期格式 | 2026年4月5日 → April 5, 2026 | April 5, 2026 → 2026年4月5日 |
| 數字格式 | 1,000,000 | 1,000,000（保持）或 100萬 |
| 標點 | 全形→半形 | 半形→全形 |
| 稱謂 | 依目標文化調整 | 依目標文化調整 |
| 頁面尺寸 | A4 → Letter（如果目標是美國市場） | Letter → A4 |
