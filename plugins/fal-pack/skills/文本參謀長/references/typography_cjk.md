## 一、核心原則（直接適用 — 中文場景仍成立）

### 1.1 Modular Scale ≥ 1.25

字級之間 ratio 太小 = 視覺層次糊在一起。階層之間至少 1.25 倍以上。

| 角色 | 字級 (pt) | 跟下一層 ratio |
|------|----------|--------------|
| Display | 24 | × 1.333 |
| Heading 1 | 18 | × 1.286 |
| Heading 2 | 14 | × 1.273 |
| Heading 3 | 11 | ✗ 1.222（< 1.25） |
| Body | 9 | — |

**問題**：Heading 3 → Body ratio 1.222 < 1.25，這兩級會糊在一起。
**修正**：Body 拉到 11pt 並把 Heading 3 提到 13pt（× 1.182 仍不夠）→ 改 Heading 3 為 14pt 跟 Heading 2 合併、或 Body 降到 9pt（傷可讀性）。
**最佳解**：移除 Heading 3 階層，只用 Display + H1 + H2 + Body 四級。少階層比 ratio 不足好。

熱門 ratio（選一個 commit）：
- 1.25（major third）— 最保守
- 1.333（perfect fourth）— 中庸
- 1.5（perfect fifth）— 戲劇性

### 1.2 多維 Hierarchy（不只靠字級）

單靠字級拉 hierarchy = 紙頁版面變單調且易扁平。至少兩個維度疊加：

| 維度 | 強層次 | 弱層次 |
|------|-------|-------|
| 字級 | ratio ≥ 1.5 | ratio < 1.25 |
| 字重 | Bold vs Regular | Medium vs Regular |
| 顏色 | 高對比 | 同色系 |
| 位置 | 頂部/左側 | 底部/右側 |
| 留白 | 周圍呼吸 | 緊貼其他元素 |

**最佳實踐**：標題用「字級放大 + 字重加粗 + 上方多留 12pt」三維度同時動。

### 1.3 字型數量 ≤ 3

一份文件最多三套字型，通常兩套就夠：
- 一套 Latin（Times New Roman / Helvetica / Cormorant Garamond / ...）
- 一套 CJK（思源宋體 / 微軟正黑體 / 思源黑體 / 標楷體）

> **反例**：「思源黑體 + 思源宋體 + Cormorant Garamond + Times New Roman + 微軟正黑體」五套字型 = 視覺風格分裂、印象散漫。

### 1.4 段落呼吸：縮排 OR 段距，**只能擇一**

| 場景 | 策略 |
|------|------|
| 書籍 / 編輯 / 文學 | 首行縮排 2 字，段距 0 |
| 商業 / 數位文件 / 簡報 | 段距 6-12pt，首行不縮排 |
| 學術論文 | 依規範（多用首行縮排，段距 0） |

🔴 **絕對禁止**：兩種同時用 = 段落間距太鬆 + 首行又縮 = 紙頁失去重量感、像草稿。

### 1.5 Squint Test（瞇眼測試）

產出後做的視覺驗收工法。**詳見第六章「Squint Test 工法」**。一句話：模糊處理後還能不能一眼看出層次。

### 1.6 ALL-CAPS 加字距（限西文場景）

英文標題 ALL-CAPS 顯示時，字母太擠。Word 設定字元間距加 0.5-1.2pt（約 0.05-0.12 em）。

🔴 中文沒有 caps 概念，本條僅用於混排英文標題或 logo 文字。

## 二、中文場景必改寫

impeccable 原則用西文邏輯設計，中文場景以下三條必改寫。

### 2.1 行距：中文 1.75-2.0 倍（不是西文的 1.5）

**為什麼**：中文字身比西文滿——同樣字級下中文字符面積約 1.4 倍於西文，垂直需要更多透氣空間。

| 場景 | 行距倍數 | 等於 Pt（11pt 內文）|
|------|---------|------------------|
| 學術 / 書籍長文 | 1.75 | 19.25pt |
| 商業文件標準 | 1.5 | 16.5pt |
| 報告 / 簡報密集 | 1.3 | 14.3pt |
| 信函 / 簡短 | 1.15 | 12.65pt |

🔴 **絕對禁止**：中文用 1.0（單倍行距）= 字疊在一起，閱讀痛苦。

### 2.2 行寬：中文每行 28-35 字（不是西文的 65ch）

**為什麼**：CJK 字元等寬，可直接算字數。
- 28 字以下 = 太擠 / 太多換行 / 視線回掃太頻繁
- 35 字以上 = 視線回掃距離過長 / 容易看錯行
- 28-35 字 = 中文閱讀舒適區

**A4 標準排版實算**：
- 紙寬 21cm，標準邊距左右各 3.17cm → 14.66cm 排版區
- 11pt 字 ≈ 0.39cm/字 → 約 37 字/行（**已超舒適區**）
- **修正**：邊距加大到左右 4cm → 13cm 排版區 → 33 字/行（剛好）
- **或**：字級拉到 12pt → 0.42cm/字 → 31 字/行（剛好）

雙欄排版時每欄寬度約 6cm → 約 15 字/行（短行密排場景，OK）。

### 2.3 Vertical Rhythm：段前 + 段後 + 行距 = 整數倍 Pt

**為什麼**：所有垂直間距共用同一個基礎單位 = 視覺自帶數學節奏感（潛意識舒適）。

選一個基礎單位（**4pt / 6pt / 8pt 任選一個 commit**），全文垂直間距全是這個單位的整數倍。

**範例（基礎 6pt）**：
- 行距 = 18pt（內文 11pt × 1.5 ≈ 16.5pt → 取 18pt）
- 段後 = 6pt 或 12pt
- 標題 1 段前 = 24pt，段後 = 12pt
- 標題 2 段前 = 18pt，段後 = 6pt
- 圖片上下 = 12pt
- 表格上下 = 12pt

🔴 **絕對禁止**：段距用 7pt / 11pt / 13pt / 17pt 等非基礎單位整數倍 = 視覺節奏破碎，潛意識會覺得不舒服但說不出為什麼。

> **附**：4pt 系統（4 / 8 / 12 / 16 / 24 / 32 / 48 / 64pt）來自 impeccable spatial-design 借鑑，但中文 Word 場景偏好 6pt 基礎單位（Word 內建段距預設多用 6 的倍數，相容性高）。

## 三、Word 中文場景必砍（impeccable 有但用不上）

| impeccable 原則 | 為什麼砍 |
|----------------|---------|
| `font-display: swap` / `preload` / FOUT/FOIT | Word 沒字型動態載入問題 |
| `clamp(min, vw, max)` fluid type | Word 不是響應式排版 |
| Container queries | Word 沒容器查詢 |
| Variable fonts / `font-optical-sizing` | Word 不支援 OpenType variation |
| `font-feature-settings: tnum / lnum / smcp` | Word 中文場景幾乎用不到 OpenType features |
| `text-wrap: balance / pretty` | Word 沒這 API（CSS-only） |
| ARIA / 觸控 target 44px | 紙本/PDF 沒互動 |

> 砍掉的不是知識本身，是**搬到 Word 中文場景沒實質效益**。需要時可回頭參照 impeccable 原檔。

## 四、中文獨有設計判斷（impeccable 西文沒寫的）

> **本節只談設計動機與視覺後果**——程式怎麼設定（CJK 字型 API、避頭尾 XML、pangu code、標點壓縮 XML）的實作見 `references/C_工程產出.md` § CJK 排版規則。本檔跟 C_工程產出.md **互不重複**（見 § 七分工聲明）。

### 4.1 為什麼必須 CJK 雙字型（Latin + eastAsia）

**根因**：Word 預設 Latin 字型（Calibri / Times New Roman）不含 CJK 字符 → 中文字 fallback 到系統預設（多為新細明體，視覺老氣難看）。

**設計後果**：必須同時設定 Latin 字型（給英文/數字）+ eastAsia 字型（給中文）。每個 run 都要——這既是工程紀律也是設計紀律。

只設一邊 = 中文字用錯字型 = 整份文件設計再講究都白搭。

🔴 實作見 `references/C_工程產出.md` § CJK 排版規則 — 字型設定。

### 4.2 為什麼避頭尾（Kinsoku）

**根因**：中文標點是全形寬度。不啟用避頭尾時：
- 句末標點（`） 、 。 ， ！ ？ 」 』`）可能掉到下一行行首 → 像「半逗號開頭」一樣突兀
- 開引號（`（ 「 『 【 〈 《`）可能停在行末 → 跟接下來內容斷開

**行首禁止字元**：`） 、 。 ， ！ ？ 」 』 】 〉 》 ） ； ：`
**行末禁止字元**：`（ 「 『 【 〈 《 （`

**設計後果**：中文出版的視覺潔淨度。Word 預設已啟用，這條知識在 review docx 時用來判斷有沒有跑掉。

🔴 實作確認見 `references/C_工程產出.md` § CJK 排版規則 — 避頭尾。

### 4.3 為什麼中英間距（pangu）

**根因**：中文跟英文/數字直接相連視覺擠壓——「中文content混排test」字符密度過高、語系邊界模糊。

**設計修正**：自動加 1/4 全形空格 → 「中文 content 混排 test」，呼吸感大幅提升、語系邊界清楚。

**呼應**：四重校對的 pangu 層在這裡接線——校對閘門會自動補空格。

🔴 實作（pangu library + OOXML autoSpaceDE/DN）見 `references/C_工程產出.md` § CJK 排版規則 — 中英混排間距。

### 4.4 為什麼標點壓縮（Overflow Punct）

**根因**：中文連續標點（例如「。」「、」「』」）每個都佔全形寬度 → 視覺累積空白、浪費版面。

**設計修正**：啟用標點壓縮，連續標點的視覺寬度自動縮減。

🔴 實作見 `references/C_工程產出.md` § CJK 排版規則 — 標點壓縮。

### 4.5 首行縮排傳統 vs 商業習慣

| 傳統 | 場景 | 縮排量 |
|------|------|-------|
| 中文書籍 / 文學 | 首行縮排 2 字 | 約 0.74cm（11pt 字 × 2 字 / 厘米換算）|
| 中文商業文件 | 首行不縮排 + 段距 | 段距 6-12pt |
| 中文學術論文 | 首行縮排 2 字 + 段距 0 | 同書籍 |
| 中文公文 | 首行縮排 2 字 + 段距 0 | 同書籍 |

🔴 **不可混用**（呼應 § 1.4）。

## 五、字型推薦（中文 Word 常用）

### 5.1 內文字型

| 字型 | 風格 | 適用 |
|------|------|------|
| 思源宋體 / Noto Serif CJK TC | 書卷氣、有重量、有襯線 | 書籍、論文、文學、傳統文件 |
| 微軟正黑體 | 中性、現代、無襯線 | 商業文件、報告、簡報、數位 |
| 思源黑體 / Noto Sans CJK TC | 開源、跨平台、無襯線 | 對外發送、避免字型缺失情境 |
| 標楷體 | 傳統、書法感 | 法律、政府公文、傳統正式 |
| 蘋方繁 / PingFang TC | macOS 預設、無襯線 | 跨 Apple 生態 |

### 5.2 字型配對原則

**最簡：一套字型多種字重**
標題用 Bold + 內文用 Regular 比兩套字型乾淨。

**經典配對：CJK 思源宋體 + Latin Times New Roman**
書籍類最經典。兩套都是襯線體，氣質統一。

**現代配對：CJK 思源黑體 + Latin Helvetica / Inter**
商業現代感。兩套都是無襯線，乾淨。

**展示配對：CJK 思源宋體 標題 + CJK 思源黑體 內文**
單一語系內襯線 / 無襯線對比，常見於雜誌封面 / 報告封面。

🔴 **絕對禁止**：兩套相似但不同的 CJK 字型（如思源黑體 + 微軟正黑體）= 視覺衝突沒層次，讀者潛意識會覺得「字型有問題」但說不出哪裡不對。

## 六、Squint Test 工法（產出後驗收）

### 6.1 為什麼要做

**讀者第一眼看到頁面時，眼睛是失焦的**——不是逐字讀，是先掃整體 hierarchy。如果模糊看不到 hierarchy，讀者會迷路、放棄、或誤解重點順序。

Squint Test = 模擬讀者第一眼，**驗證 hierarchy 真的有出來**。

### 6.2 操作步驟

1. **產出 .docx 後用 Word COM 轉 PDF**：
   ```python
   doc.ExportAsFixedFormat(pdf_path, 17)  # wdExportFormatPDF
   ```

2. **PDF 轉 PNG（72 DPI 模糊預覽用）**：
   ```bash
   pdftoppm input.pdf output -png -r 72
   ```

3. **PNG 高斯模糊 5px**：
   ```python
   from PIL import Image, ImageFilter
   img = Image.open('output-1.png')
   blurred = img.filter(ImageFilter.GaussianBlur(5))
   blurred.save('output-1-blurred.png')
   ```

4. **表弟讀模糊版 PNG**，自答三題：
   - 最重要的元素能一眼看出嗎？
   - 第二重要的呢？
   - 區塊分組清楚嗎？

### 6.3 通過標準

三題全 Yes = 通過。任一題模糊看不出 = hierarchy 失敗，回鍛打。

### 6.4 替代簡易版

沒 Pillow / pdftoppm 環境時：
- Word 內建縮放到 25% 預覽 → 等於 4× 縮小 → 模糊感等同
- 或：截圖手機相機鏡頭離螢幕 1 公尺看
- 或：眼鏡度數不夠的人脫眼鏡看（不要笑，真的有用）
