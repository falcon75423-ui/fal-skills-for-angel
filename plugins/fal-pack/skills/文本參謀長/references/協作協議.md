# 跨 Skill 協作協議（出版商模式）

> 讀取時機：被其他 Skill 委託時載入。觸發詞：「請出版商來處理」「交給出版商」「出版商」。

---

## 核心隱喻

**表弟 = 出版商。內容方 = 作者。**

出版商不是印刷廠。出版商拿到書稿之後：
1. 先讀到能跟作者深度對談的程度
2. 帶著自己的專業視角提出呈現方案
3. 與作者討論，共同決定最好的呈現方式
4. 執行，產出頂級品質的成品
5. 校樣確認——確認沒有扭曲作者的意思

**重大呈現決策共同拍板，排版品質表弟全權負責。**

---

## 協作流程

```
內容方（作者）                          表弟（出版商）
    │                                     │
    ├── 產出內容                            │
    ├── 填寫 Brief ──────────────────→ 收到 Brief
    │                                     ├── 讀稿（理解內容+讀者+意圖）
    │                                     ├── 產出「呈現方案」
    │                                ←──── 排版對話：提案 + 取捨說明
    ├── 回饋 / 調整 ─────────────────→ │
    │    （可來回多次）                      ├── 共同決議
    │                                     ├── 執行（付印）
    │                                     ├── 驗證管線
    │                                ←──── 視覺校樣（PDF/PNG）
    ├── 確認 / 要求修改 ────────────→ │
    │                                     ├── 修改（如需要）
    │                                ←──── 最終交付 .docx
    └── 收貨                               └── 完成
```

## Brief 格式

### 三級路由

Brief 的級別決定表弟走哪條路徑：

**輕量 Brief → 套版路徑**
```
【委託類型】套版
【模板名稱】[templates/ 中的模板名稱]
【資料】[JSON / YAML / 結構化資料]
【輸出路徑】[期望的檔案路徑]
```
無需排版對話，直接填充產出。

**標準 Brief → 排版路徑**
```
【委託類型】排版
【文件類型】法律 / 商業 / 學術 / 報告 / 其他
【讀者是誰】[一句話描述讀者和閱讀情境]
【內容】[完整內容或摘要]
【Design System】（可選）[指定 Design System 或使用預設]
【特殊要求】（可選）[任何特殊偏好或限制]
```
需要排版對話，表弟理解內容後提出方案。

**重型 Brief → 起草/審稿路徑**
```
【委託類型】起草 / 審稿 / 翻譯
【文件類型】[同上]
【讀者是誰】[同上]
【內容】[完整內容]
【內容結構契約】[結構規格的路徑或內容]
【品質期望】[具體的品質標準]
【特殊要求】（可選）
```
需要深度讀稿 + 排版對話 + 完整品質把關。

### 通用三欄位（所有 Brief 必備）

依照生態系版圖的通用 Brief 格式：
```
【做什麼】一句話說清楚要委託的事
【為什麼】這個產出拿來做什麼
【回傳什麼】期望的交付物格式（.docx / .pdf / 兩者）
```

## 內容結構契約

> 當一個 Skill 的交付物有固定結構（如占星師的六層報告），內容方需要提供「內容結構契約」讓表弟知道骨架長什麼樣。

### 契約格式

```yaml
# content_contract.yaml
name: "占星師個人盤報告"
version: "1.0"
owner: "占星師"
last_updated: "2026-04-05"

structure:
  - level: 1
    name: "封面"
    content_type: "fixed"
    description: "報告標題、受測者姓名、日期"
    
  - level: 1
    name: "第一層：基礎資料"
    content_type: "structured"
    description: "類型、策略、權威、定義、輪廓"
    sub_sections:
      - name: "類型"
        content_type: "text_block"
      - name: "策略"
        content_type: "text_block"
        
  - level: 1
    name: "第二層：通道分析"
    content_type: "repeating"
    description: "每條通道獨立分析"
    repeat_unit:
      - name: "通道名稱"
      - name: "三頻譜描述"
      - name: "個人化解讀"

metadata:
  total_layers: 6
  estimated_pages: "15-25"
  primary_language: "zh-TW"
```

### 契約的生命週期

1. **建立**：工頭鍛造 Skill 時，與 使用者 共同定義
2. **存放**：存在表弟的 `templates/contracts/` 目錄
3. **使用**：被委託時，表弟讀取契約了解骨架
4. **更新**：內容方結構變動時，通知表弟更新契約和對應模板
5. **版本化**：每次變動記錄版本號，模板跟著更新

---

## 模板所有權

**原則：模板存在表弟，契約存在表弟，內容方持有契約的更新權。**

```
文本參謀長/
├── templates/
│   ├── contracts/           # 內容結構契約
│   │   └── 占星師_個人盤.yaml
│   ├── 占星師_個人盤/     # 對應的模板檔案
│   │   ├── template.docx    # Word 模板
│   │   └── design_system.py # Design System 定義
│   └── ...
```

**為什麼不存在內容方**：
- 內容方不懂排版，沒有能力維護模板
- 排版的演化由表弟主導（比如發現某個間距在長報告中效果不好）
- 單一來源原則：模板只有一份權威版本

**內容方的權利**：
- 更新內容結構契約（結構變了要通知）
- 在排版對話中提出偏好和否決
- 確認視覺校樣（內容層次的否決權）

---

## 排版對話

> 這是出版商模式的核心——不是單向交付，是雙向對話。

### 對話發生的時機

- 標準 Brief 和重型 Brief 必有排版對話
- 輕量 Brief（套版）跳過
- 但如果表弟在套版過程中發現模板不適合這份內容，有權升級為排版對話

### 對話的形式

表弟讀完稿件後，產出「呈現方案」：

```
## 呈現方案

### 我讀到的
[一段話描述：內容核心訊息、讀者是誰、情感基調]

### 我建議的呈現方式
[具體的視覺設計方向，用視覺語言描述]

### 我的理由
[為什麼這樣呈現最適合這份內容和這個讀者]

### 需要你確認的決策
- [ ] [決策1]
- [ ] [決策2]

### 替代方案
[如果你不同意，還有這些選項...]
```

---

## 視覺校樣

### 校樣格式

排版完成後，表弟產出：
1. **PDF 預覽**（Word COM → PDF）
2. **關鍵頁面 PNG**（封面、目錄頁、內容首頁、典型內頁）
3. **校樣說明**：指出哪些是刻意的設計決策，哪些可以調整

### 內容方的確認範圍

✅ 可以否決：
- 「這個層次太不顯眼，讀者會錯過」（內容層次）
- 「這裡的順序不對，應該先講X再講Y」（邏輯順序）
- 「你理解錯了，這段的重點不是這個」（內容理解）

❌ 不在否決範圍：
- 「我不喜歡這個字型」（視覺細節由表弟負責）
- 「行距太大/太小」（技術判斷由表弟負責）
- 「顏色換一個」（除非有品牌規範，否則表弟決定）

## 首次協作流程

> 第一次跟某個 Skill 合作時，沒有契約也沒有模板。以下是從零開始的流程：

1. **接收 Brief** — 即使沒有契約，Brief 三欄位（做什麼/為什麼/回傳什麼）不能少
2. **讀稿** — 深度理解內容，特別注意結構模式（哪些是固定的？哪些每次不同？）
3. **排版對話** — 與內容方討論呈現方案，比平常更多對話（因為沒有先例）
4. **執行 + 交付** — 正常流程
5. **🔴 結晶提議** — 交付完成後，表弟主動問：「這份文件的結構和排版要結晶為契約和模板嗎？下次就不用重來。」
   - 內容方說要 → 進入公版化流程，同時產出 `content_contract.yaml`
   - 內容方說不要 → 記錄為一次性委託，不建模板
6. **記錄** — 無論是否公版化，把這次的協作經驗加入下方「已建立的協作關係」表
