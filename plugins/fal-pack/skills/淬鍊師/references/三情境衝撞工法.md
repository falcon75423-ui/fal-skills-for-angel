# 三情境衝撞工法

> 讀取時機：淬鍊流程第 5 步「定型閘門」必讀。SKILL.md 的「四種武器配置」段也指引到這裡。
>
> 核心：**未經 subagent 衝撞 = 未淬鍊成功**。

---

## 為什麼要衝撞

從第 4 步萃取出來的智慧雛形，**自我論證已建構成形**——剛萃取智慧的人主觀判斷「我有沒有漏洞」會被自我合理化遮蔽。

> **M25 meta-level 自指悖論**：讓被污染者判斷自己污染程度 = 結構性 meta 失效。

對應 F-004 的「驗收 = subagent 無條件」原則：**淬鍊驗證也要 subagent 無條件**。

🔴 **這是淬鍊師獨立成 Skill 的根本動機之一**——讓「鍛造主體」跟「淬鍊驗證」由不同 LLM 執行，從結構上避免自指悖論。

---

## 機制：邊界 + 衝突 + 反例三類

「至少三個有張力的情境」不是隨便三個，是三類**會挑戰智慧邊界**的情境：

| 類別 | 用途 | 挑出方式 |
|------|------|---------|
| **邊界情境** | 測試「在邊緣會不會失效」 | 列出最極端但仍合法的輸入 |
| **衝突情境** | 測試「跟其他原則打架時誰贏」 | 找會跟既有閘門 / 原則同時觸發的情境 |
| **反例情境** | 測試「什麼時候不該套用」 | 找看似命中但其實不適用的情境 |

🔴 **三類缺一不可**——只跑邊界 + 衝突 = 沒驗反例 = 淬鍊未成功。

---

## 依產出層用不同驗證法

不同產出層用不同的衝撞型式——強度跟工法不同：

### 上游（世界觀 / 人設 / 原則）

**驗證方法**：哲學張力測試（subagent 帶不同立場進場 / 重型可召喚淬煉團 5 角色）

| 三情境形式 | 範例 |
|-----------|------|
| 邊界 | 原則在極端情境會不會失效？（如「永遠 X」碰到「Y 比 X 重要」的情境） |
| 衝突 | 兩條原則打架時誰贏？（如「韌性」 vs 「沉沒成本不打架」碰到正在進行的鍛造怎辦） |
| 反例 | 什麼情境下這條原則不適用？（如「YAGNI」碰到「未爆彈防護」） |

**🔴 召喚淬煉團 vs subagent 的判別**：
- 議題影響半徑 ≥ 整個 SKILL.md / 跨 Skill 通用 → **召喚淬煉團 5 角色**
- 議題影響半徑限於單一條目 / 流程 → **subagent 三情境衝撞夠用**

### 中游（閘門 / 條目 / 流程）

**驗證方法**：操作壓力測試（subagent 模擬 failure mode）

| 三情境形式 | 範例 |
|-----------|------|
| 邊界 | 訊號模糊時判別（如「閘門觸發詞」邊緣字串會不會誤觸 / 漏觸）|
| 衝突 | 跟既有閘門同時觸發（如「修改三關」+ 「白話對齊」同時要動，順序衝突嗎）|
| 反例 | 不需要套用的情境（如「拍板訊息骨架」用在純技術細節對話會不會多餘）|

### 下游（偵測機制 / 計數 / 踩雷）

**驗證方法**：失效偵測測試（subagent 製造失效訊號）

| 三情境形式 | 範例 |
|-----------|------|
| 邊界 | 訊號邊緣值（如「閘門失效追蹤」表的「復發 ≥ 2 次」邊界）|
| 衝突 | 多訊號同時（如同一案件觸發兩個閘門失效）|
| 反例 | 合法 null 場景（如某類案件本來就沒這欄資料，不該被誤判失效）|

## subagent 召喚 prompt 模板

呼叫 Agent 工具時，**prompt 必須包含**：

```
身份：你是被淬鍊的智慧的「敵人」——你的工作是找出這條智慧的漏洞，不是驗證它。

待測智慧（雛形）：
{完整貼上萃取的智慧文字}

待測智慧的設計初衷：
{淬鍊師對它的說明——它解決什麼問題、想擋什麼失效模式}

請設計三類情境衝撞並逐一檢驗：

1. 邊界情境：請給出至少 1 個極端但仍合法的輸入/場景，並判定智慧是否仍能正確運作
2. 衝突情境：請給出至少 1 個跟其他既有原則同時觸發的場景，並判定智慧的優先級是否清楚
3. 反例情境：請給出至少 1 個看似命中但其實不適用的場景，並判定智慧是否會誤套

對每一類情境：
- 描述具體場景（不要抽象論述）
- 給出智慧在該場景下的預測行為
- 判定：通過 / 失敗 / 不確定
- 失敗時：點明漏洞的層級（結構 / 規則 / 策略 / 動機）

最後給出總體判定：
- 全通過 → 淬鍊成功（給出短評）
- 至少一情境失敗 → 列出漏洞清單 + 建議補強方向

報告 ≤ 600 字。
```

🔴 **prompt 紀律**：
- 明確設定 subagent 為「敵人」（不是合作者）—— 抵抗「禮貌通過」的 LLM 傾向
- 要求**具體場景**（不要抽象論述）—— 抽象論述容易自圓其說
- 要求**判定 + 漏洞層級**—— 失敗時直接告訴你下一步往哪修
- 要求**字數上限**—— 避免 subagent 灌水稀釋核心發現

## 通過/失敗處理流程

```
三情境衝撞
  ↓
全通過 → 淬鍊成功 → 第 6 步結構性融入
某情境暴露漏洞 → 還有漏洞要繼續打磨 → 回第 4 步重新結晶 → 再次衝撞
持續失敗（≥ 3 輪打磨仍過不去）→ 重新檢視第 1-3 步是否誤判（可能根因還沒找到）
```

🔴 **不允許「應該可以」「我有信心」當作通過**——只要有一個情境 subagent 判定失敗或不確定，就是淬鍊未成功。

> **詳細收斂標準（漏洞層級下降軌跡 R1-R4 + 四輪上限）** → `references/收斂標準.md`

## 跟其他工法的關係

### 跟「四層深度透視」的關係

| 機制 | 由誰跑 | 性質 |
|------|--------|------|
| **四層深度透視** | 淬鍊師自跑 | 結構分析（解剖刀） |
| **三情境衝撞** | subagent 跑 | 動態壓力測試（淬火池） |

四層找出「可能的漏洞模式」 → 三情境衝撞**主動製造這些漏洞模式**驗證智慧能扛住。

### 跟「世界觀升級協議」的關係

世界觀升級協議五題 = **三情境衝撞驗證在「上游世界觀」層的具體版**：
- 五題的前 4 題提供結構配套（具象失效 / 上游位置 / 中游結構 / 下游偵測）
- 五題的第 5 題「回測驗證」= 三情境衝撞的單情境簡化版

當淬鍊產出落在上游世界觀時，**走五題協議 + 補做三情境衝撞**（五題第 5 題僅單情境，不夠）。

> **完整五題 checklist** → `references/世界觀升級協議.md`

### 跟「淬煉團」的關係

| 機制 | 規模 | 何時用 |
|------|------|--------|
| **subagent 三情境衝撞** | 1 個 subagent 跑三情境 | 中游淬鍊 / 下游淬鍊 / 上游低張力 |
| **淬煉團 5 角色** | 5 個角色多輪碰撞 | 上游高張力 / 跨 Skill 通用智慧 |

不衝突——淬煉團也可以**輸出三情境作為衝撞素材**，再由淬鍊師對著素材跑驗證。

> **淬煉團召喚規則** → `references/淬煉團召喚指南.md`

## 實戰範例：F-004 四輪迭代

來源：`匠神大道/鍛造記錄/2026-04-26_火神鍛造-F004-即時驗收雙簽+修繕兩種情境分類.md`

待測智慧：「修繕的兩種情境並列分類（情境 A 對外部客戶 + 情境 B 火神自身）」

| 輪 | subagent 抓到的漏洞 | 漏洞層級 | 修法 |
|---|-------------------|---------|------|
| R1 | 「情境 A」「情境 B」概念混淆——對外/對內二分法在「跨客戶 + 火神自身」案件失效 | **結構層** | 把「並列分類」改寫，加跨客戶情境 C |
| R2 | 即時驗收雙簽的觸發條件「拍板前」太模糊——「拍板」邊界在哪？ | **規則層** | 補定義：「拍板 = 使用者字面確認的時刻」 |
| R3 | 雙簽流程在「情境 B 火神自身」會自指——工頭驗收工頭 | **策略層** | 加「subagent 無條件驗收」硬規則 |
| R4 | 工頭可能繞過驗收（僥倖心理）——規則無法擋誠實度 | **動機層** | **觸底**——再加規則只會變繁文縟節，達到當前閘門設計極限 → 定型 |

四輪後達到「動機層 / 誠實度層」= 觸底 = 淬鍊成功。

> **四輪上限的 meta 風險（為什麼第五輪通常邊際歸零）** → `references/收斂標準.md`

## 工法的工法（meta 層）

三情境衝撞本身也應該被三情境衝撞——

- **邊界**：subagent 找不到反例怎辦？（已寫「召喚淬煉團」加碼）
- **衝突**：跟「淬火（八項）」的順序怎排？（已寫「淬火驗 Skill 本身、衝撞驗萃取的智慧」分工）
- **反例**：什麼時候不該跑三情境衝撞？（純改 typo / 非智慧層的單純 Edit → 不需要衝撞）

**結論**：本工法三情境自審通過，可投入使用。
