# 收斂標準

> 讀取時機：跑三情境衝撞驗證時，判斷「夠不夠了」。
>
> 核心：**淬鍊不是越多輪越好，是知道什麼時候停**。

---

## 為什麼需要收斂標準

沒有收斂標準會發生什麼：
- subagent 每輪都能再找到一個漏洞 → 看起來每輪都「有進步」 → 永遠停不下來
- 第 N+1 輪抓到的漏洞層級反而比第 N 輪更淺 → 已經邊際效益歸零還在加規則
- 加到第 6/7 輪規則之間開始矛盾 → 機制本身變成 M28 反例（「再驗證一輪小補丁」退化迴圈）

🔴 **淬鍊驗證機制本身需要停損點，否則機制本身就會變成它要擋的失效模式**。

---

## 漏洞層級下降軌跡

來源：F-004 第一個完整四輪驗證實例（M28 + 世界觀條文「對齊是必經流程」皆在四輪內定型）。

| 輪次 | 抓到的漏洞層級 | 性質 | 修法 |
|------|--------------|------|------|
| **R1** | **結構層** | 情境本身的盲點 / 假設不成立 / 概念混淆 | 重新定義概念邊界 |
| **R2** | **規則層** | 邊緣情況未處理 / 警示力不足 | 補規則 + 加閘門 |
| **R3** | **策略層** | 偷渡路徑 / 模糊訊號 / 大量產出格式 | 補檢查機制 + 加 fail-fast |
| **R4** | **動機層** | 執行者誠實度，規則無法擋的領域 | **觸底——加規則只會變繁文縟節** |

🔴 **規律**：好的迭代會讓漏洞層級**逐輪變深**。第 N+1 輪抓到的層級比第 N 輪淺 = 警報（可能根因還沒找到，往回查）。

---

## 三個收斂訊號（任一觸發即定型）

### 訊號 1：層級觸底

漏洞落到「動機/誠實度層」= 達到當前閘門設計極限 = 定型。

**判別**：subagent 抓到的失效模式是「執行者明知規則但選擇繞過 / 僥倖 / 偷懶」，**且該失效模式無法靠任何規則額外條款擋住**——只能靠執行者自己誠實。

**舉例**：F-004 第 4 輪抓到「工頭可能繞過驗收」——這不是規則漏洞，是誠實度問題。再加「禁止繞過」的規則沒用，因為要繞過的人本來就會無視規則。

🔴 **觸底 = 定型**：再加規則只會稀釋規則的整體警示力（規則太多 = 沒人記得）。

### 訊號 2：跨閘門互補

殘留的漏洞**屬於姊妹閘門的職責**——不在當前閘門加規則。

**判別**：subagent 抓到的失效模式，已被既有的另一個閘門 / 規則 / 條目擋住。當前閘門加規則 = 重複工作 = 一進一出原則違反。

**舉例**：F-004 第 4 輪 M28 殘留「context 延伸時要對齊」的某個邊界 case，但該 case 已落在 M26「首次失效根因強制升級」範圍內 → 不在 M28 加規則，留給 M26 處理。

🔴 **跨閘門互補 = 定型**：每個閘門守自己的範圍，不要互相重疊。

### 訊號 3：四輪上限（硬規則）

第五輪通常邊際效益歸零或產生反規則衝突——**硬上限**。

**為什麼是 4 不是 5**：
- 第 1 輪：抓結構層（最深）
- 第 2 輪：抓規則層（中等深）
- 第 3 輪：抓策略層（中淺）
- 第 4 輪：抓動機層（最淺，已觸底）
- 第 5 輪：要嘛重複前 4 輪（無新意）、要嘛抓到無關範圍（規則打架）

實證：F-001 / F-002 / F-004 都在 4 輪內定型。N=3 累積樣本支持「4 輪足夠」假設。

🔴 **若 4 輪後仍有殘留 → 不是繼續加規則，而是重新檢視第 1-3 步是否誤判**——可能整個淬鍊方向錯了，根因還沒找到。

## 收斂判斷流程

```
跑完一輪三情境衝撞
  ↓
有失敗情境？
  ├─ 否 → 全通過 → 淬鍊成功 → 第 6 步結構性融入
  └─ 是 → 失敗的漏洞屬哪一層？
            ├─ R1/R2/R3 → 修 → 再衝撞下一輪
            └─ R4 動機層 → 觸底 → 定型 → 進第 6 步
                          ├─ 殘留屬姊妹閘門職責 → 跨閘門互補 → 定型
                          └─ 已第 4 輪 → 四輪上限 → 強制定型
```

---

## 跨閘門互補的判別

殘留漏洞屬不屬於姊妹閘門？三個判別問題：

1. **這個漏洞的根因在我的範圍內嗎？**——不在 = 不該我管
2. **既有閘門能不能擋住這個漏洞？**——能 = 不該我管
3. **如果我加規則擋這個漏洞，會不會跟既有閘門重疊？**——會 = 一進一出違反

三個都答「是」 → 不在當前閘門加規則，標註為「跨閘門互補殘留」 → 定型。

## 落地實例：F-004 四輪迭代

來源：`匠神大道/鍛造記錄/2026-04-26_火神鍛造-F004-即時驗收雙簽+修繕兩種情境分類.md`

待測智慧：「修繕的兩種情境並列分類（情境 A 對外部客戶 + 情境 B 火神自身）」

| 輪 | subagent 抓到的漏洞 | 漏洞層級 | 處理 |
|---|-------------------|---------|------|
| **R1** | 「情境 A」「情境 B」概念混淆——對外/對內二分法在「跨客戶 + 火神自身」案件失效 | **結構層** | 修：把「並列分類」改寫為「兩種情境並列總綱 + 跨客戶情境 C」 |
| **R2** | 即時驗收雙簽的觸發條件「拍板前」太模糊——「拍板」邊界在哪？ | **規則層** | 修：補定義「拍板 = 使用者字面確認的時刻」 |
| **R3** | 雙簽流程在「情境 B 火神自身」會自指——工頭驗收工頭 | **策略層** | 修：加「subagent 無條件驗收」硬規則 |
| **R4** | 工頭可能繞過驗收（僥倖心理）——規則無法擋誠實度 | **動機層 觸底** | **定型**——再加規則只會變繁文縟節 |

四輪後達到「動機層 / 誠實度層」= 觸底 = 淬鍊成功。

**總計**：4 輪、4 層、3 次修補（R1-R3 各修一次）+ 1 次定型判斷（R4）。

---

## 為什麼觸底就停

如果第 4 輪後繼續：

- **第 5 輪可能加規則**：「禁止繞過驗收」「驗收必須由 subagent 跑」「subagent 報告必須附評分」⋯⋯ → 規則堆積 = 反淬鍊（信條 3 違反）
- **第 5 輪可能加流程**：「驗收前必須跑 X」「驗收後必須記 Y」⋯⋯ → 流程膨脹 = 摩擦力增加（信條 6 違反）
- **第 5 輪可能加文件**：「在 SKILL.md 增寫 Z 段提醒」⋯⋯ → 規則層補丁（M02 層級錯配必失效）

🔴 **關鍵領悟**：規則無法擋誠實度。**設計上承認這個邊界**，把誠實度交還給執行者，比假裝靠規則能擋有用。

呼應淬鍊師信條 1：**做減法不做加法**——觸底時做的減法是「不再加規則」。

---

## 收斂後的後續行動

定型 = 進入第 6 步「結構性融入」（三階段十步流程）。具體：

1. 把通過驗證的智慧寫進對應位置（依產出層分流——上游 SKILL.md / 中游 lessons / 下游偵測表）
2. 跑一進一出檢查（新增什麼 → 退役什麼）
3. 跑觸發脈絡回寫（更新 description / 觸發詞）
4. 把 R1-R4 的漏洞發現寫進 Bridge Document（給現場主人持有）

---

## 工法的工法（meta 層）

收斂標準本身的收斂——

- **R1 自審**：4 層的順序對嗎？（已驗證——F-001/F-002/F-004 都符合「越往後越淺」規律）
- **R2 自審**：4 輪上限會不會有例外？（允許例外——「重新檢視第 1-3 步」是 fallback）
- **R3 自審**：跨閘門互補會被當作懶得修的藉口嗎？（三個判別問題擋住）
- **R4 自審**：使用者真的會用這個收斂標準嗎？（淬鍊師 dogfood + F-004 實例 = 證據）

**結論**：收斂標準 R1-R4 自審通過，可投入使用。
