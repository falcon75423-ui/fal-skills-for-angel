# 觸發脈絡回寫工法

> 讀取時機：每次淬鍊結束時的「結構固化」階段第 10 步。也適用：現場主人 SKILL.md description / 觸發詞需要更新時。
>
> 核心：**Description 是寫給模型看的觸發訊號，不是寫給人看的摘要**。

---

## 為什麼這件事重要

Skill 最大的失效模式之一不是內容寫不好，是**該觸發時沒觸發**。

從不被觸發的 Skill = 不存在的 Skill。即使內容是金子，沒被召喚就是死的。

🔴 **核心領悟**：description 字段的閱讀者是 LLM 不是人類。寫法應該優化「LLM 看到什麼字面會聯想到此 Skill」，不是優化「人類讀完知道這 Skill 在幹嘛」。

---

## 觸發脈絡的四個維度

每次回寫要檢視的四個維度：

| # | 維度 | 寫在哪 | 性質 |
|---|------|--------|------|
| 1 | **Description（系統訊號）** | SKILL.md frontmatter `description` | 模型每次決定召喚與否時讀的字面 |
| 2 | **動詞觸發詞** | SKILL.md「觸發與啟動」段 | 使用者主動發起的動作詞 |
| 3 | **對象觸發詞** | SKILL.md「觸發與啟動」段 | 隨物件出現的觸發詞（如「淬煉這個 md」）|
| 4 | **反觸發** | SKILL.md「觸發與啟動」段 | 名詞引用不觸發（如「鍛造完跑了淬鍊」描述歷史不召喚）|

🔴 **缺一不可**——只有 description 沒觸發詞 = 觸發模糊；只有觸發詞沒反觸發 = 過度觸發污染對話。

---

## 維度 1：Description

### 寫法：兩段結構

```
{身份/個性 + 1 句靈魂}—— {功能描述 + 動詞動作}。觸發：「{動詞觸發 1}」「{對象觸發 2}」「{動詞觸發 3}」⋯⋯
```

範例（淬鍊師）：

```
淬鍊師 ⚒️ 冷眼工匠 — 把已驗證的工匠技術從場景綁定中解放出來，對既有智慧資本（Skill / md / 案件經驗 / 客戶能力）做審計與提煉。觸發：「淬煉這個 X」「召喚淬鍊師」「淬煉這次經驗」「淬煉 賽事分析 的判斷力」「淬煉這個 SKILL.md」「淬煉這份 memory」「淬煉客戶能力」
```

### 紀律

🔴 **「觸發：」段是核心**——寧可冗長也不要漏。模型看到字面會立刻聯想，這比身份段重要。

🔴 **包含具體例子的觸發詞**：「淬煉這個 SKILL.md」勝過「淬煉檔案」（具體名 > 泛稱）。

🔴 **避免被另一個 Skill 偷走觸發**：寫前掃一遍其他 Skill 的 description，找重疊的字眼，調整使你的更精準。

---

## 維度 2：動詞觸發詞

使用者**主動發起**的動作詞。

### 設計原則

| 原則 | 範例 |
|------|------|
| 用搜尋者語言不用設計者語言（M07）| 「召喚淬鍊師」（搜尋者）勝過「啟動淬鍊流程」（設計者） |
| 動詞 + 對象（不要孤動詞）| 「淬煉這次經驗」勝過「淬煉」 |
| 涵蓋常用同義詞 | 「淬煉」「淬鍊」（簡繁體都收）|
| 包含「召喚」「請」「讓」開頭的句式 | 「召喚 X」「請 X 進場」「讓 X 來看 Y」 |

### 範例（淬鍊師）

| 觸發詞 | 觸發場景 |
|--------|---------|
| 召喚淬鍊師 | 直接召喚 |
| 召喚冷眼工匠 | 用綽號召喚 |
| 淬煉 X | 動詞主動 |
| 淬煉這次經驗 | 案件後 |
| 淬煉 賽事分析 的判斷力 | 客戶能力 |

---

## 維度 3：對象觸發詞

隨**物件**自然出現的觸發詞——使用者不一定要說「召喚 X」，看到物件本身就該觸發。

### 範例（淬鍊師）

| 觸發詞 | 觸發場景 |
|--------|---------|
| 淬煉這個 md | 拿到一個 md 檔要精煉 |
| 淬煉這個 SKILL.md | 拿到一個 SKILL.md 要審計 |
| 淬煉這份 memory | 拿到 MEMORY.md 要清理 |
| 淬煉客戶能力 | 對某個客戶 Skill 整體做審計 |

🔴 **對象觸發是隱式 → 必須寫明在 description**——模型看不到的 = 不存在。

---

## 維度 4：反觸發（名詞引用不觸發）

### 為什麼需要

名詞引用**描述歷史 / 提到該 Skill** ≠ 召喚該 Skill。
過度觸發 = 對話被 Skill 不必要地接管 = 摩擦力增加。

### 範例（淬鍊師）

| ❌ 不觸發 | 為什麼 |
|----------|--------|
| 「鍛造完跑了淬鍊」 | 描述歷史 |
| 「我覺得這部分淬鍊得不錯」 | 評論既有產出 |
| 「淬鍊師上次的工法」 | 引用而非召喚 |

### 寫法

在「觸發與啟動」段加一行：

```
❌ **名詞引用不觸發**：「鍛造完跑了淬鍊」（描述歷史不召喚）
```

---

## 回顧的時機

🔴 **每次淬鍊結束**——不是只在「description 看起來舊了」時。理由：
- 每次淬鍊都可能改了該 Skill 的某個能力 → 觸發脈絡可能不準
- 每次都檢視 = 五分鐘工作 + 防止漂移
- 等到「看起來舊了」才改 = 已經錯過很多本該觸發的場景

也適用：
- 鍛造主體完工時
- 淬鍊發現「這個 Skill 經常該被召喚但沒被召喚」時
- 跨 Skill 通用智慧浮現，影響多個 Skill 的觸發域時

---

## 回顧的五步驟

```
1. 拉出本次更動了什麼能力 / 範圍 / 工法
2. 對照當前 description——這些更動有反映在 description 嗎？
3. 對照當前觸發詞列表——有沒有新的觸發詞要加？
4. 對照當前反觸發——有沒有新的「名詞引用」場景要排除？
5. 跑形式檢查：description 字面是否包含本次更動的關鍵字？
```

### 步驟 5 的形式檢查

| 檢查 | 通過信號 |
|------|---------|
| 字面包含核心動詞 | description 有「{動詞} X」字串 |
| 字面包含核心對象 | description 有「淬煉 X」「處理 X」「審計 X」字串 |
| 字面包含「召喚 + 角色名」 | description 有「召喚 X」字串 |
| 觸發詞 ≥ 4 個 | 通常 ≥ 4 才足以涵蓋多種召喚句式 |

任一不通過 → 補強 description。

## 實戰案例：淬鍊師 description 設計

### v1（不夠 pushy，會漏觸發）

```
淬鍊師 — 對既有智慧資本做審計與提煉。
```

問題：
- 沒「觸發：」段 → 模型不知道什麼時候召喚
- 「智慧資本」太抽象 → 模型不會聯想到「md / Skill / 經驗」
- 沒身份個性 → 跟其他 Skill 沒區分度

### v2（加觸發詞但仍不夠具體）

```
淬鍊師 — 對既有智慧資本做審計與提煉。觸發：「淬煉」
```

問題：
- 「淬煉」單詞 = 沒對象 = 邊界模糊（會跟「淬煉鋼鐵」這種無關詞觸發）
- 沒列出具體的對象觸發

### v3（pushy 不漏觸發）

```
淬鍊師 ⚒️ 冷眼工匠 — 把已驗證的工匠技術從場景綁定中解放出來，對既有智慧資本（Skill / md / 案件經驗 / 客戶能力）做審計與提煉。觸發：「淬煉這個 X」「召喚淬鍊師」「淬煉這次經驗」「淬煉 賽事分析 的判斷力」「淬煉這個 SKILL.md」「淬煉這份 memory」「淬煉客戶能力」
```

通過：
- 身份段含個性「冷眼工匠」+ 靈魂句「把已驗證的工匠技術從場景綁定中解放出來」
- 「智慧資本」括號展開為「Skill / md / 案件經驗 / 客戶能力」(具體名 > 泛稱)
- 7 個觸發詞涵蓋多種召喚句式（動詞 + 對象 + 召喚 + 角色名 + 客戶名 + 物件型）
- 動詞觸發 + 對象觸發 + 召喚句 三類並存

## 過度觸發的處理

如果某個 Skill 太常被誤觸（描述太 pushy）：

| 訊號 | 處理 |
|------|------|
| 對話經常出現「我以為你會召喚 X 但其實不需要」 | 該 Skill 觸發詞太寬，加反觸發 |
| 觸發後使用者立刻說「不是這個意思」 | 觸發詞跟其他 Skill 重疊，調整觸發詞使更精準 |
| 觸發後對話 5 分鐘內被另一個 Skill 接管 | 該 Skill 是「過渡」性質，不應該獨立觸發 |

---

## 工法的工法（meta 層）

觸發脈絡回寫工法本身——

- **R1 自審**：4 個維度涵蓋了所有觸發場景嗎？（description + 動詞 + 對象 + 反觸發 = 完整覆蓋）
- **R2 自審**：5 步驟回顧能扛住「忘記跑」嗎？（不能——靠淬鍊師紀律「每次淬鍊結束必跑」）
- **R3 自審**：會不會被當作「最後 5 分鐘隨便補一下」？（風險存在，靠 5 步驟的形式紀律）
- **R4 自審**：淬鍊師會用這個工法嗎？（每次淬鍊結束都會用 = 高頻使用）

**結論**：本工法 R1-R4 自審通過，可投入使用。
