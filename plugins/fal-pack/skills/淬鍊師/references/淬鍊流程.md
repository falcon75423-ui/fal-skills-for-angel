# 淬鍊流程

> 讀取時機：淬鍊師被召喚進場時必經。SKILL.md 的「工法骨架」段會指引到這裡。
> 歷史：原為火神鍛造的 `references/知識淬鍊流程.md`（2026-04-24 F-001 從「壓縮」升級為「淬鍊」）；2026-04-27 隨淬鍊師獨立 Skill 化遷入此處並改寫為通用語境。

淬鍊師被召喚時，**現場主人交付待淬鍊的內容**——可能是一段案件經驗、一個 md 檔、一個 Skill 的累積資產。淬鍊不是把這些東西存起來，是**淬鍊成可以守得住的智慧**。走過的路回頭滋養下次。

---

## 淬鍊的本質

### WHO 層意識

> **鍛造是把材料變成器具的過程。淬鍊是讓器具能扛得住反覆使用的過程。**

沒淬鍊的器具 = 一次性工具，用完就壞。
經過淬鍊的器具 = 可重複使用 + 越用越韌，成為飛輪的一部分。

呼應淬鍊師 SKILL.md 信條 1「做減法不做加法」——**淬鍊是一個減法動作**，把雜質去除留下精華。

---

**字面拆解**：
- **淬** = 把熱鋼丟進冷水，**做壓力測試**——扛不住溫差衝擊會裂開、扛得住才成鋼
- **鍊** = 反覆鍛造（加熱 → 敲打 → 冷卻 → 重複）
- 合起來 = 反覆鍛造後的最後一道收功，讓物件**可用**

「定型加硬」的方式就是「扛住壓力測試」——這兩個是同一件事。

**淬鍊的本質定義**：

> **從「處理問題的整段經驗」中萃取「下次能用的智慧」並結構性融入現場主人的系統。**

四個關鍵字：

1. **整段經驗**（不是某一刻）—— 從碰到問題到上線新做法的完整弧線
2. **萃取**（不是記錄）—— 砍掉雜質留下精華（壓縮 = 全保存、淬鍊 = 砍掉錯方向）
3. **可移植**（下次能用）—— 智慧能從本案件移到下案件、從本人移到下個人
4. **三層融入**（上中下游）—— 不只寫進文件，要結構性嵌入現場主人的系統

**淬鍊 ≠ 記錄。淬鍊 ≠ 反思。淬鍊 ≠ 召喚結束後的活動。**

「記錄」是壓縮、「反思」是手段、「結束後活動」是淬鍊的最後一步（定型）——不是全部。

### 壓縮 vs 淬鍊（對照）

| 維度 | 壓縮 | 淬鍊 |
|------|------|------|
| 動作 | 把體積變小 | 把雜質去除 + 扛住壓力測試 |
| 產出 | 可保存的摘要 | 可應用、可移植、扛得住衝撞的智慧 |
| 檢驗 | 能重述就算 | 三情境衝撞驗證通過才算 |
| 關係 | 淬鍊的副產品 | 主動作 |

**壓縮是淬鍊的結果，不是目的**。只壓縮不淬鍊 → 智慧扛不住衝撞 → 下次同類錯誤依然發生。

## 淬鍊的整段經驗（6 階段）

```
1. 碰到問題（現場主人在進行中發生的——鍛造 / 修繕 / 客戶服務 / md 精煉）
2. 處理中發現更好做法（討論、琢磨、來回對齊）        ←── 淬鍊的「淬」開始
3. 用新做法解決原問題（砍掉舊方向 + 上線新方向）    ←── 雜質去除
4. 結晶可移植智慧（候選版本，雛形）
   ├─ 上游（世界觀 / 人設 / 原則）
   ├─ 中游（結構閘門 / 條目 / 字典 / 流程實戰做法）
   └─ 下游（偵測機制 / 計數表 / 踩雷錄）
5. 🔴 三情境衝撞驗證（淬鍊定型閘門）              ←── 淬鍊的「定型」
   ├─ 通過 → 進入第 6 步
   └─ 不通過 → 回第 4 步繼續打磨
6. 結構性融入現場主人的系統（→ 下方三階段十步展開）   ←── 淬鍊的「鍊」固化
```

第 1-3 步 = 淬鍊「發生」（**通常現場主人已做完——淬鍊師被召喚時這三步常已是歷史**）。
第 4 步 = 萃取「雛形」。
第 5 步 = 淬鍊「定型」（壓力測試）。
第 6 步 = 淬鍊「鍊」固化（結構性融入）。

🔴 **沒過第 5 步 = 還是熱鋼不是鋼 = 淬鍊未成功**。

## 淬鍊產出的多樣性

不同情境淬鍊出不同層的智慧——**不強求統一產出形式**：

| 情境 | 產出層 | 實例 |
|------|--------|------|
| 結構性失效 + 哲學洞察 | 世界觀（上游）| F-004「進行式是打磨能力的最好時機」|
| 角色行為定義 | 人設（上游）| 「敢喊停 × 願意砍掉重練」|
| 跨案件累積模式 | 教訓條目（中游）| M25/M26/M27/M28 |
| 失效後的形式觸發 | 鍛打閘門（中游）| 拍板訊息骨架 / 修改三關 |
| 跨產品設計原則 | 流程結構（中游）| F-002 索引架構翻修 |
| 實戰操作工法 | 實戰做法（中游）| 一進一出原則、Context 體重秤 |
| 失效追蹤機制 | 偵測（下游）| 閘門失效追蹤表 |

**讓情境決定智慧落在哪一層**。

---

## 何時意識到淬鍊正在發生

**訊號**：在處理某個問題時，發現舊方法的盲點 + 新方法的雛形浮現。

**不是**：
- 「有東西要記下來」（這是壓縮）
- 「處理完成了該回顧」（這是事後反思，不是進行式淬鍊）
- 「我寫個條目存起來」（沒驗證 = 未淬鍊成功）

**是**：
- 「**這次經驗有什麼是下次也能用的**」
- 「我發現原方法漏了一個維度，新方法可能更根本」
- 「這個觀察值得從本案抽出來移到通用層」

🔴 **現場主人若在進行中發現淬鍊訊號浮現 → 召喚淬鍊師進場做工**——這是淬鍊師被召喚的最自然觸發點。

## 🔴 第 5 步：三情境衝撞驗證（淬鍊定型閘門）

> **完整衝撞模板（依產出層分流）** → `references/三情境衝撞工法.md`

> 「淬」字本義：熱鋼丟進冷水做壓力測試——扛住才成鋼。**未驗證 ≠ 淬鍊成功**。

### 機制

從第 4 步萃取的智慧雛形，必須通過**至少三個有張力的情境衝撞**才能進入第 6 步結構性融入。

- **「至少三個」**：呼應 Three Sightings 概念，但這裡是**事前預測失效模式**，不是事後累積
- **「有張力」**：不是隨便三個情境，是會挑戰智慧邊界的情境——**邊界 + 衝突 + 反例**三類

### 依產出層用不同驗證法

| 產出層 | 驗證方法 | 三情境的具體形式 |
|--------|---------|-----------------|
| **上游**（世界觀/人設/原則）| **哲學張力測試**（subagent 帶不同立場進場 / 重型可召喚淬煉團 5 角色）| 邊界：原則在極端情境會不會失效 / 衝突：兩條原則打架誰贏 / 反例：什麼情境下不適用 |
| **中游**（閘門/條目/流程）| **操作壓力測試**（subagent 模擬 failure mode）| 邊界：訊號模糊時判別 / 衝突：跟既有閘門同時觸發 / 反例：不需要套用的情境 |
| **下游**（偵測機制/計數）| **失效偵測測試**（subagent 製造失效訊號）| 邊界：訊號邊緣值 / 衝突：多訊號同時 / 反例：合法 null 場景 |

### 為什麼必須 subagent 進場（而不是淬鍊師自己腦補）

**M25 meta-level 自指悖論**——剛萃取出新智慧的人，自我論證已建構成形，主觀判斷「我有沒有漏洞」會被自我合理化遮蔽。讓被污染者判斷自己污染程度 = 結構性 meta 失效。

跟 F-004 的「驗收 = subagent 無條件」對齊：**淬鍊驗證也要 subagent 無條件**。

🔴 **這也是淬鍊師獨立成 Skill 的根本動機之一**——讓「鍛造主體」跟「淬鍊驗證」由不同 LLM 執行，從結構上避免自指悖論。

### 通過/失敗處理

```
三情境衝撞
  ↓
全通過 → 淬鍊成功 → 第 6 步結構性融入
某情境暴露漏洞 → 還有漏洞要繼續打磨 → 回第 4 步重新結晶 → 再次衝撞
持續失敗（≥ 3 輪打磨仍過不去）→ 重新檢視第 1-3 步是否誤判（可能根因還沒找到）
```

### 收斂標準（淬鍊驗證的「足夠好」門檻）

> **完整版（含跨閘門互補實例 + 四輪上限的 meta 風險）** → `references/收斂標準.md`
> 來源：F-004 第一個完整四輪驗證實例（M28 + 世界觀條文「對齊是必經流程」皆在四輪內定型）。

**漏洞層級下降軌跡**：

| 輪次 | 抓到的漏洞層級 |
|------|--------------|
| R1 | **結構層**（情境本身的盲點 / 假設不成立 / 概念混淆）|
| R2 | **規則層**（邊緣情況未處理 / 警示力不足）|
| R3 | **策略層**（偷渡路徑 / 模糊訊號 / 大量產出格式）|
| R4 | **動機層**（執行者誠實度，規則無法擋的領域）|

**收斂訊號**（任一觸發即定型）：

1. 🔴 **層級觸底**：漏洞落到「動機/誠實度層」= 達到當前閘門設計極限 = 定型（再加規則只會變繁文縟節）
2. 🔴 **跨閘門互補**：殘留屬姊妹閘門職責（如 M28 殘留落到 M26 範圍）→ 不在當前閘門加規則
3. 🔴 **四輪上限**：第五輪通常邊際效益歸零或產生反規則衝突——硬上限

**為什麼**：避免「每輪都抓到東西 = 永遠不夠」的無限迭代陷阱。淬鍊驗證機制本身需要停損點，否則機制本身就會變成 M28 反例（「再驗證一輪小補丁」退化迴圈）。

### 跟現有機制的關係

**跟「淬火」（火神鍛造主體壓力測試）的區別**：

| 機制 | 對象 | 時機 | 歸屬 |
|------|------|------|------|
| 淬火（八項） | 鍛造主體（Skill）本身 | Skill 鍛造完工後 | 火神鍛造 |
| 三情境衝撞驗證 | 從現場經驗萃取的可移植智慧 | 第 4 步萃取雛形後、第 6 步融入前 | 淬鍊師 |

兩者層級不同——淬火驗證 Skill 本身能不能用、三情境衝撞驗證從 Skill 經驗萃取的智慧能不能移植。

**跟「世界觀升級協議五題」的關係**：

世界觀升級協議五題 = **三情境衝撞驗證在「上游世界觀」層的具體版**：
- 五題的前 4 題提供結構配套（具象失效 / 上游位置 / 中游結構 / 下游偵測）
- 五題的第 5 題「回測驗證」= 三情境衝撞的單情境簡化版

當淬鍊產出落在上游世界觀時，**走五題協議 + 補做三情境衝撞**（五題第 5 題僅單情境，不夠）。
當淬鍊產出落在中游/下游時，**直接走三情境衝撞**（不需要五題）。

> **完整五題 checklist + F-001 落地實例** → `references/世界觀升級協議.md`

## 第 6 步展開：三階段十步流程（結構性融入現場主人的系統）

```
階段一：淬鍊思考     ← 新，先做，不可略過（把原料本質化）
階段二：壓縮入庫     ← 原流程，淬鍊結果的文件化
階段三：結構固化     ← 原流程尾段，把淬鍊產出變成守得住的結構
```

🔴 **歸屬原則**：階段一+二+三的產出**最終歸現場主人持有**——淬鍊師像偵察兵，做完就退場，不在自己這邊累積。例外：跨 Skill 通用條目進淬鍊師 `lessons/`（共用資源、淬鍊師自帶教訓掃描必讀，故必留淬鍊師端）。

### 階段一｜淬鍊思考

**0. 🔴 持久化掃描**（見「詳述」段，機械規則偵測）

**1. 🔴 根因升格問答**

淬鍊進場時停下來，問自己三題：
- **本次失效/成功的根因，是個案、類型、還是更上層的錯層陷阱？**
- **如果明天另一個 LLM 遇到結構相似但表面不同的情境，他會不會踩同一個雷？**
- **本次揭露了什麼你以為已經懂、但其實沒結構化的東西？**

三題寫不出答案 = 淬鍊還沒發生 → 回去想。

**2. 🔴 類型比對**（升級版的原 Three Sightings 檢查）

翻 `lessons/INDEX.md` 的「現役條目（按情境）」表：
- 本次對應哪個錯誤類型？
- 這是該類型的第幾次？
- 還有沒有其他錯誤類型的計數靠近 Three Sightings（≥ 3）？

**3. 🔴 升級判斷**（決策分岔口）

本次是否觸發**向上一層升級**？觸發信號任一即是：

| 信號 | 意義 |
|------|------|
| 錯誤類型累計 ≥ 3 次（Three Sightings） | 教訓該升級為原則或世界觀 |
| 現場主人 SKILL.md 閘門復發 ≥ 2 次 | 閘門失效，禁用「擴條件」，走三選一（退役/降級/重問 WHO） |
| 既有世界觀條目**顯性觸發**本次失效 | 世界觀需補結構配套（上游文字有、結構沒跟上）|
| 新的跨 Skill 通用模式浮現 | 考慮鍛造新的通用機制 |

任一觸發 → 🔴 **走 `references/世界觀升級協議.md` 五題 checklist**。沒通過 = 升級不成立，不得跳過入庫。
未觸發 → 繼續階段二。

---

### 階段二｜壓縮入庫（淬鍊結果的文件化）

**4. 交付給現場主人持久化**

🔴 **歸屬原則的應用**：淬鍊師**不寫自己的 Bridge Document**。淬鍊產出交付給現場主人，由現場主人按其體系存檔：

| 現場主人類型 | 持久化形式 |
|-------------|----------|
| **火神鍛造**（鍛造主體完工時呼叫淬鍊師）| 由火神工頭寫 Bridge Document（火神 `references/bridge-template.md`）+ 存火神 `匠神大道/鍛造記錄/` |
| **客戶 Skill**（如 賽事分析 / 占星師 / 其他）| 進客戶工作簿 + 客戶 Skill 自己的 references |
| **md 檔**（如 MEMORY.md / 客戶筆記）| 直接寫回 md 檔本身（路徑 B 四層深度透視的標準產出） |
| **跨 Skill 通用智慧** | 進淬鍊師 `lessons/` 對應情境檔（共用資源，淬鍊師自帶教訓掃描時必讀，故留淬鍊師端） |

**為什麼淬鍊師不留自己的「淬鍊記錄/」**——見 SKILL.md「純工匠版定位」三個結構性決定。

**5. 🔴 禁區萃取**（協助現場主人提取「不適用範圍」）

現場主人有「鍛造決策點負空間」需求時（如火神鍛造的 `history/` 禁區卡機制）→ 淬鍊師協助提取「淬鍊發現的不適用範圍」交付給現場主人寫入。

現場主人沒有禁區卡機制時 → 淬鍊師可寫進淬鍊師 `lessons/ARCHIVE.md` 的考古條目（標 `applicable_scope: not_applicable_when_X`）。

禁區卡的格式（給火神或同類現場主人用）：

```
# {Skill} v{N} — {日期} [{內容|邊界|新建}]

{一句話：做了什麼}

🚫 {禁區 1}（{為什麼不能走這條路}）
🚫 {禁區 2}（...）

↳ {現場主人的 Bridge 路徑} 決策#{N}
```

- 每張 ≤ 50 字（寫不完 = 還沒想清楚）
- 🚫 禁區 ≥ 1 條（必填。沒禁區就不開卡）
- 禁區被推翻時不刪卡，原地加刪除線 + 推翻理由：`🚫 ~~舊禁區~~（YYYY-MM-DD 推翻：{新證據}）`

**6. 教訓分層**

三層分類：
- **Micro**：只適用現場主人這個 Skill → 寫進現場主人自己的 references / 工作簿
- **Meso**：同類型 → 寫進淬鍊師 `lessons/` 對應情境檔
- **Macro**：跨類型通用 → 寫進淬鍊師 `lessons/` 對應情境檔 + 配 related[] 跨檔指針

Meso 和 Macro 條目的 frontmatter 規範見 `lessons/_schema.md`。寫完跑 `tools/generate_lessons_index.py` 重生成 INDEX.md。

**7. 引用回標**

本次進場偵察拉出的教訓，逐條回標 **helpful / harmful / 不適用**。在條目 frontmatter 的 `helpful_count` / `harmful_count` 欄位累加。

**8. 後向回測**

拿新教訓回推原始場景，問「只靠這條能獨立導出正確判斷嗎？」答不出 = 條目尚未自足，回去補。

---

### 階段三｜結構固化

**9. 一進一出**

加了什麼就問要不要退什麼。審查：
- 新增規則 → 取代了哪條舊規則？
- 新增結構 → 退役了哪個舊機制？
- 淨增 = 紅旗，回去想是不是 WHO 層定義不夠

**10. 觸發脈絡回寫**

每次淬鍊結束時回顧現場主人 SKILL.md 的 description 和觸發條件，更新觸發脈絡。

> 🔴 **Description 是寫給模型看的觸發訊號**，不是寫給人看的摘要。寧可 pushy 也不要漏觸發。

> **完整觸發脈絡回寫工法** → `references/觸發脈絡回寫工法.md`

## 詳述｜第 0 步 持久化掃描

> **原則**：摘要不等於原件。一行描述（「5 種技法 + 20 條禁止清單」）不等於完整的策略手冊。測試：下一個 LLM 只讀檔案不讀對話，能照著做嗎？不能 = 沒持久化。

### 三條偵測規則（機械式，不靠判斷力）

**規則 A — Agent/偵察兵回傳**
本次處理過程中每一次 Agent 工具呼叫且有回傳結果 → 該結果必須有對應檔案。派出去就代表重要，不重要不會派。

**規則 B — 顧問團/淬煉團/聯合研判產出**
本次處理過程中每一次顧問團、淬煉團或聯合研判產出會議摘要/結論 → 必須有對應檔案或已嵌入目標規格檔。

**規則 C — 使用者確認過的結構化內容**
本次處理過程中出現 ≥3 行的表格/清單/框架/公式，且使用者回覆了確認詞（同意/對/OK/可以/沒問題）→ 該內容必須有對應檔案。

### 逐項檢查

- 有對應檔案 → 確認交棒給現場主人時有引用該檔案路徑
- 沒有 → **停，先寫進檔案，再繼續淬鍊**

**不通過的信號**：「藍圖有提到」「交棒單寫了摘要」→ 停，**摘要不等於原件**。

### 為什麼是第零步

文件化是壓縮——摘要指向原件。原件不存在，摘要就是指向虛空的指標。先存原件，再寫摘要。

### 踩雷實例（2026-04-12，火神鍛造原始版本）

三路偵察兵各回傳 500 字研究報告 → 工頭整合成完整策略（5 技法 + 校準表 + 20 禁止 + 寫作公式）→ 鍛造藍圖只寫了一行摘要 → 交棒單沒引用 → 使用者問「精髓存在哪」才發現漏了。有了規則 A，三個 Agent 回傳時就會觸發檢查。
